1. O que é um banco de dados?

Um banco de dados é uma coleção organizada de informações relacionadas, estruturadas de forma que possam ser facilmente acessadas, gerenciadas e atualizadas. A palavra-chave aqui é organização: um arquivo de texto com mil linhas de dados pode conter informação, mas não é um banco de dados — falta a estrutura que permite recuperar exatamente o que você precisa em milissegundos.

Pense no Instagram: cada foto que você posta precisa ser associada ao seu perfil, ter uma lista de curtidas, comentários, tags e metadados como data e localização. Quando alguém abre o feed, o aplicativo busca em frações de segundo as publicações mais recentes de centenas de pessoas que você segue, classifica por relevância e exibe com imagens em alta resolução. Isso só é possível porque todos esses dados estão armazenados em bancos de dados cuidadosamente projetados. O mesmo raciocínio vale para um banco financeiro que precisa garantir que uma transferência debite de uma conta e credite em outra de forma atômica — sem que nenhuma das duas operações fique pela metade — ou para um e-commerce que precisa checar em tempo real se um produto está disponível em estoque antes de confirmar um pedido.

Antes dos bancos de dados, as aplicações armazenavam dados em arquivos no sistema de arquivos do sistema operacional. Esse modelo causava problemas sérios: redundância (os mesmos dados duplicados em múltiplos arquivos), inconsistência (versões diferentes do mesmo dado), dificuldade de acesso concorrente (dois programas tentando escrever no mesmo arquivo ao mesmo tempo) e ausência de segurança granular. Os bancos de dados foram criados exatamente para resolver esses problemas, fornecendo uma camada de abstração entre os dados e as aplicações que os consomem.

Definição formal

Segundo Ramakrishnan e Gehrke (Database Management Systems, 3ª ed.), um banco de dados é "uma coleção de dados que descrevem os fatos de um minimundo" — ou seja, um recorte do mundo real relevante para uma determinada organização ou aplicação.

Evolução do armazenamento de dados

Para entender por que precisamos de bancos de dados, é instrutivo olhar para a evolução das formas de armazenar informação em sistemas computacionais. Cada salto resolveu uma limitação concreta da geração anterior — e cada solução trouxe trade-offs próprios. Compreender esse histórico ajuda a escolher a ferramenta certa para o problema atual e a evitar a tentação de "improvisar com arquivos" quando o caso pede um SGBD de verdade.

Quando ainda faz sentido um arquivo .txt ou .csv?

Para dados imutáveis, pequenos e processados em lote (logs históricos, exports semanais, configurações estáticas), arquivos ainda são apropriados. O ponto de virada é quando você precisa de consultas concorrentes, integridade referencial ou consultas seletivas eficientes. Nesse momento, mover para um SGBD economiza muito mais código de validação do que custa em complexidade operacional.

2. SGBD — Sistema Gerenciador de Banco de Dados

É fundamental distinguir duas coisas que frequentemente se confundem: o banco de dados (os dados em si, os arquivos físicos em disco) e o SGBD — Sistema Gerenciador de Banco de Dados. O SGBD é o software que gerencia o banco: ele recebe comandos, verifica permissões, garante a integridade dos dados, controla o acesso concorrente de múltiplos usuários ou aplicações, gerencia memória e cache, faz backups e recuperação em caso de falha. O banco de dados em si são os arquivos físicos no disco — sem o SGBD, você não tem como interagir com eles de forma segura e eficiente.

Um SGBD cumpre quatro funções essenciais: definição de dados (permite criar e alterar a estrutura das tabelas via DDL), manipulação de dados (permite inserir, atualizar, deletar e consultar via DML e DQL), controle de dados (gerencia permissões de acesso via DCL) e controle de transações (garante as propriedades ACID via TCL). Essas quatro funções correspondem exatamente às cinco sublinguagens do SQL, que estudaremos mais adiante neste material.

O mercado oferece vários SGBDs, cada um com características distintas. PostgreSQL é o mais robusto e completo entre os open-source, com suporte a tipos avançados (JSON nativo, arrays, geometria), extensões e conformidade rigorosa com o padrão SQL — será nossa ferramenta principal neste módulo. MySQL (e seu fork MariaDB) é amplamente usado em aplicações web de médio porte. SQLite é uma biblioteca embarcada, sem servidor separado, ideal para apps mobile e desktop. MongoDB é um banco de dados NoSQL orientado a documentos JSON, útil quando o esquema é altamente variável. A escolha do SGBD é um RNF da categoria Restrições de Design e deve ser justificada, não aleatória.

Por que PostgreSQL neste módulo?

PostgreSQL é open-source, gratuito, tem uma comunidade enorme, segue fielmente o padrão SQL ANSI, suporta JSON nativamente, possui tipos avançados e extensões como PostGIS para dados geográficos. Aprender PostgreSQL prepara você para migrar facilmente para MySQL ou qualquer outro banco relacional — e para entender o que diferencia os bancos NoSQL dos relacionais.

Como o SGBD garante integridade

Uma das responsabilidades mais importantes do SGBD é garantir que os dados permaneçam íntegros — consistentes com as regras do negócio. Isso é feito por meio de constraints (restrições) definidas no esquema, que o SGBD verifica a cada operação de escrita. Se uma constraint for violada, a operação é rejeitada com um erro antes de qualquer dado ser persistido. Esse mecanismo evita que a responsabilidade de validação fique 100% na camada da aplicação — que pode ter bugs ou ser bypassada diretamente via SQL por alguém com acesso ao banco.

Pipeline interno: o que acontece quando você executa uma query

Quando sua aplicação envia SELECT * FROM usuarios WHERE id = 1; ao banco, o SGBD não interpreta o texto literalmente — ele passa a query por uma sequência de estágios bem definidos. Compreender esse pipeline torna explicável por que algumas queries são rápidas e outras não, e o que tuning de banco realmente significa.

EstágioO que fazSaída
1. ParserLê o texto SQL, valida sintaxe e converte em uma árvore de análise (AST).Erro de sintaxe ou árvore válida
2. Analyzer / RewriterResolve nomes de tabelas e colunas, aplica regras (views, RLS), checa permissões DCL.Plano lógico
3. Planner / OptimizerAvalia múltiplos planos de execução (sequential scan, index scan, hash join, nested loop) e escolhe o de menor custo estimado com base em estatísticas das tabelas.Plano físico
4. ExecutorExecuta o plano físico em pipeline, lendo páginas do disco (ou do cache em memória), aplicando filtros e devolvendo linhas.Resultado para o cliente
5. Storage layerLê e escreve páginas de 8 KB (no PostgreSQL) em arquivos no disco. O WAL (Write-Ahead Log) registra cada alteração antes de tocar a tabela.Dados persistidos com durabilidade

Ao escrever (INSERT, UPDATE, DELETE), o SGBD primeiro grava a operação no WAL — Write-Ahead Log e só depois aplica a mudança nas páginas de dados. Esse log sequencial é o que garante a Durabilidade do ACID: se a máquina cair logo após o COMMIT, ao reiniciar o SGBD relê o WAL e reaplica as operações que ainda não tinham chegado às páginas em disco. Backups incrementais e replicação física também usam o WAL como fonte da verdade — por isso ele é central tanto para confiabilidade quanto para alta disponibilidade.

Investigando o plano: EXPLAIN

No PostgreSQL, o comando EXPLAIN ANALYZE SELECT ... mostra o plano físico escolhido pelo planner e o tempo real de cada operação. É a primeira ferramenta de tuning de query: se você ver Seq Scan em uma tabela grande sendo filtrada por uma coluna, falta um índice ali.

3. Modelo Entidade-Relacionamento (ER)

Antes de escrever uma linha de SQL, precisamos modelar o problema. O Modelo Entidade-Relacionamento (ER), proposto por Peter Chen em 1976, é a linguagem gráfica padrão para representar a estrutura de informação de um sistema. Ele é independente de tecnologia: um diagrama ER pode ser implementado em PostgreSQL, MySQL, Oracle ou qualquer outro banco relacional. O modelo ER nos força a pensar sobre o que armazenar antes de pensar como.

Um diagrama ER tem três elementos fundamentais. Entidades são os "substantivos" do sistema — as coisas sobre as quais precisamos armazenar informação. Exemplos: Cliente, Produto, Pedido, Funcionário. Cada entidade tem atributos — as propriedades que descrevem cada instância dessa entidade. Um Cliente pode ter nome, email e telefone. Um Produto pode ter nome, preço e estoque. Um atributo especial é a chave primária (PK) — um atributo (ou conjunto de atributos) cujo valor identifica unicamente cada instância da entidade. Finalmente, relacionamentos são as associações entre entidades — os "verbos" do sistema. Um Cliente realiza Pedidos. Um Pedido contém Produtos.

A cardinalidade especifica quantas instâncias de uma entidade podem se relacionar com instâncias de outra. Os três tipos fundamentais são: 1:1 (um para um) — cada pessoa tem exatamente um CPF; 1:N (um para muitos) — um cliente pode ter vários pedidos, mas cada pedido pertence a um único cliente; e N:M (muitos para muitos) — um pedido pode conter vários produtos, e um produto pode aparecer em vários pedidos. Relacionamentos N:M exigem uma tabela intermediária (tabela associativa ou junction table) para serem implementados no modelo relacional.

A regra do "quadrado em volta do losango" — relacionamentos N:M

Quando o relacionamento é muitos-para-muitos (N:M) e carrega atributos próprios — por exemplo, quantidade e preco_unit em "PEDIDO contém PRODUTO" —, a notação de Chen desenha um retângulo tracejado em volta do losango. Isso sinaliza que esse relacionamento, ao ser implementado no modelo lógico, vai virar uma tabela associativa com chave primária composta. Atributos do relacionamento (como quantidade) são desenhados como elipses ligadas ao losango.

Modelagem em 3 etapas: Conceitual, Lógico, Físico

Modelar um banco de dados não é uma atividade única — é um processo de refinamento progressivo do O QUE existe no domínio até o COMO implementar em um SGBD específico. A literatura clássica organiza esse processo em três níveis sucessivos, cada um com um artefato próprio e uma audiência distinta.

📦 1. Modelo Conceitual — O QUE

Artefato: diagrama ER (notação de Chen ou Crow's Foot).
Foco: entidades, atributos e relacionamentos do domínio. Independente de SGBD, linguagem de programação ou tecnologia.
Audiência: stakeholders de negócio, product owners, especialistas do domínio.
Pergunta-chave: "Quais são as coisas que o sistema precisa lembrar e como elas se relacionam?"

🗂️ 2. Modelo Lógico — COMO

Artefato: esquema relacional com tabelas, colunas, tipos genéricos, PKs e FKs.
Foco: tradução das entidades em tabelas, dos relacionamentos 1:N em FKs e dos N:M em tabelas associativas. Ainda independente do SGBD escolhido.
Audiência: engenheiros de software, DBAs, arquitetos de dados.
Pergunta-chave: "Como organizo essas informações em tabelas para que sejam consistentes e consultáveis?"

⚙️ 3. Modelo Físico — EM QUAL BANCO

Artefato: SQL DDL específico do SGBD — CREATE TABLE, índices, partições, tablespaces.
Foco: tipos concretos do SGBD (SERIAL vs AUTO_INCREMENT vs IDENTITY), JSONB, índices, particionamento, tuning específico.
Audiência: desenvolvedores que vão escrever migrations e DBAs.
Pergunta-chave: "Que código SQL exato vou rodar no PostgreSQL 16 (ou no banco de produção)?"

Por que separar as 3 etapas?

A separação evita acoplar decisões de domínio (que são duradouras) com decisões de tecnologia (que mudam). Se amanhã você precisar migrar de PostgreSQL para Oracle, o modelo conceitual e o lógico permanecem válidos — só o físico precisa ser reescrito. Inverter a ordem (codar primeiro o SQL, "modelar" depois) é um anti-padrão clássico que produz esquemas tortos e migrations dolorosas.

Por que modelar antes de criar tabelas?

Modelar em ER antes de escrever DDL tem dois benefícios principais. Primeiro, permite identificar problemas de design — entidades esquecidas, relacionamentos ambíguos, atributos mal posicionados — antes que qualquer código seja escrito. Corrigir um diagrama leva minutos; corrigir um banco de dados em produção com dados reais pode levar horas e exigir migrações complexas. Segundo, o diagrama ER é uma linguagem comum entre técnicos e não técnicos: um gerente de produto consegue entender e validar um diagrama ER mesmo sem saber SQL.

4. Modelo Relacional — Tabelas, PK e FK

O modelo relacional, proposto por Edgar F. Codd em 1970, é o modelo de dados mais amplamente usado no mundo. Toda informação é organizada em tabelas (também chamadas de relações, em homenagem ao modelo matemático de álgebra relacional). Cada tabela tem um nome único no banco, um conjunto fixo de colunas (atributos) com tipos de dados definidos, e um conjunto de linhas (tuplas), onde cada linha representa uma instância da entidade.

A chave primária (Primary Key, PK) é uma coluna (ou conjunto de colunas) que identifica unicamente cada linha da tabela. Toda tabela deve ter uma PK. O SGBD cria automaticamente um índice na PK, tornando buscas por esse campo extremamente rápidas. Em PostgreSQL, o tipo SERIAL (ou BIGSERIAL para tabelas maiores) é a forma mais comum de criar PKs numéricas auto-incrementais. Para sistemas que precisam de IDs globalmente únicos sem coordenação central, usa-se UUID.

A chave estrangeira (Foreign Key, FK) é uma coluna que referencia a PK de outra tabela, estabelecendo o relacionamento entre elas no nível físico. Quando você define uma FK no SGBD, ele passa a verificar automaticamente a integridade referencial: você não pode inserir uma linha com cliente_id = 99 em pedidos se não existir um cliente com id = 99 na tabela clientes. Também não pode deletar um cliente que ainda tem pedidos associados — a menos que configure um comportamento de cascata (ON DELETE CASCADE).

Da ER às tabelas — regras formais de tradução

A passagem do modelo conceitual (ER) para o modelo lógico (relacional) segue um conjunto compacto de regras mecânicas. Aplicá-las disciplinadamente evita os dois erros mais comuns de iniciantes: (1) esquecer a tabela associativa em N:M e (2) colocar a FK do lado errado em 1:N. Memorize essas quatro regras — elas resolvem 95% dos casos práticos.

Construção ERTradução para o modelo relacionalExemplo
EntidadeVira uma tabela com mesmo nome (no plural, em snake_case).USUARIO → tabela usuarios
AtributoVira uma coluna com tipo apropriado. O atributo identificador vira PRIMARY KEY.email → coluna email VARCHAR(200)
Relacionamento 1:1Vira uma FK em uma das duas tabelas (geralmente na obrigatória), marcada com UNIQUE para impedir duplicação.Pessoa ↔ CPF: pessoas.cpf_id INTEGER UNIQUE REFERENCES cpfs(id)

SQL — exemplo completo: as 4 tabelas do diagrama ER USUARIO/PEDIDO/PRODUTO 
-- Entidade USUARIO → tabela usuarios
CREATE TABLE usuarios (
  id     SERIAL       PRIMARY KEY,
  nome   VARCHAR(100) NOT NULL,
  email  VARCHAR(200) UNIQUE NOT NULL
);

-- Entidade PRODUTO → tabela produtos
CREATE TABLE produtos (
  id     SERIAL          PRIMARY KEY,
  nome   VARCHAR(100)    NOT NULL,
  preco  NUMERIC(10,2)    NOT NULL
);

-- Relacionamento 1:N "USUARIO faz PEDIDO" → FK no lado N (pedidos)
CREATE TABLE pedidos (
  id          SERIAL    PRIMARY KEY,
  data        TIMESTAMP DEFAULT NOW(),
  usuario_id  INTEGER   NOT NULL REFERENCES usuarios(id)
);

-- Relacionamento N:M "PEDIDO contém PRODUTO" → tabela associativa
-- PK composta = (pedido_id, produto_id) — ambas FKs
CREATE TABLE pedido_produto (
  pedido_id   INTEGER NOT NULL REFERENCES pedidos(id),
  produto_id  INTEGER NOT NULL REFERENCES produtos(id),
  quantidade  INTEGER NOT NULL CHECK (quantidade > 0),
  PRIMARY KEY (pedido_id, produto_id)
);
Por que a PK composta na tabela associativa?

A combinação (pedido_id, produto_id) é única por definição: o pedido 42 só pode conter o produto 7 uma vez — se quiser duas unidades, isso vira quantidade = 2. Marcar a PK como composta impede acidentalmente inserir a mesma combinação duplicada e ainda cria o índice composto que acelera buscas em ambas as direções.

Normalização — organizando para evitar redundância

A normalização é um processo de organizar as tabelas de um banco relacional para eliminar redundância e dependências anômalas. Cada forma normal estabelece um conjunto de regras que a tabela deve obedecer. As três primeiras são as mais importantes na prática:

1FN — Primeira Forma Normal

Regra: todos os atributos devem ser atômicos (sem listas ou grupos repetidos numa mesma coluna) e deve haver uma PK.
Exemplo de violação: coluna telefones com o valor "(11)9999-0001, (11)9999-0002" — dois valores numa coluna só.
Solução: criar uma tabela separada telefones(id, cliente_id, numero).

2FN — Segunda Forma Normal

Regra: a tabela deve estar em 1FN e todos os atributos não-chave devem depender funcionalmente de toda a PK. Relevante quando a PK é composta.
Exemplo de violação: tabela item_pedido(pedido_id, produto_id, nome_produto) — nome_produto depende só de produto_id, não de pedido_id.
Solução: mover nome_produto para a tabela produtos.

3FN — Terceira Forma Normal

Regra: a tabela deve estar em 2FN e não deve haver dependências transitivas (atributo não-chave dependendo de outro atributo não-chave).
Exemplo de violação: tabela pedidos(id, cliente_id, cidade_cliente, estado_cliente) — estado_cliente depende de cidade_cliente, que depende de cliente_id.
Solução: mover dados do cliente para a tabela clientes e referenciar por FK.

Desnormalização controlada

Em alguns casos de alta performance (relatórios analíticos, data warehouses), intencionalmente desnormalizamos — repetimos dados — para evitar JOINs custosos em consultas de leitura. Essa é uma decisão de trade-off consciente, documentada como RNF de Desempenho, e não um erro de design. Normalize primeiro; desnormalize com evidência de necessidade.

5. SQL — As 5 Sublinguagens

SQL (Structured Query Language) não é uma linguagem monolítica — ela é composta por cinco sublinguagens, cada uma com um propósito distinto. Essa organização existe porque diferentes papéis interagem com o banco de maneiras diferentes: um DBA gerencia estrutura e permissões; um desenvolvedor escreve DML e DQL; um sistema de monitoramento opera transações via TCL. Entender as sublinguagens é entender a separação de responsabilidades no SQL.

Cada sublinguagem opera em um nível diferente de abstração. DDL opera no nível do esquema — a estrutura que contém os dados. DML e DQL operam no nível dos dados em si — as linhas dentro das tabelas. DCL opera no nível das permissões — quem pode fazer o quê. TCL opera no nível das transações — garantindo que grupos de operações sejam executados de forma atômica. Um profissional de banco de dados competente domina todas as cinco sublinguagens e sabe quando usar cada uma.

É importante notar que alguns comandos DDL (como TRUNCATE) e os comandos DCL têm efeito imediato e implícito de COMMIT em muitos SGBDs — ou seja, não podem ser revertidos com ROLLBACK. Já os comandos DML fazem parte de transações e podem ser revertidos até o COMMIT ser executado. Essa diferença tem implicações práticas importantes: você pode corrigir um UPDATE errado com ROLLBACK, mas não pode desfazer um DROP TABLE da mesma forma.

DDL
Data Definition Language
Define e altera a estrutura do banco: cria, modifica e remove tabelas, índices e outros objetos. Comandos DDL têm efeito imediato e implicitamente fazem COMMIT.
CREATE · ALTER · DROP · TRUNCATE
DML
Data Manipulation Language
Manipula os dados dentro das estruturas existentes: insere, atualiza e deleta linhas. Faz parte de transações e pode ser revertido com ROLLBACK.
INSERT · UPDATE · DELETE · MERGE
DQL
Data Query Language
Consulta dados sem modificá-los. Alguns autores incluem SELECT dentro do DML, mas a separação é didaticamente útil: SELECT tem sintaxe própria e riquíssima.
SELECT
DCL
Data Control Language
Controla permissões de acesso: concede e revoga privilégios a usuários e roles. Garante que cada usuário acesse apenas o que precisa — princípio do menor privilégio.
GRANT · REVOKE
TCL
Transaction Control Language
Gerencia transações: delimita blocos de operações que devem ser executados de forma atômica. Se qualquer operação falhar, todas são revertidas.
BEGIN · COMMIT · ROLLBACK · SAVEPOINT

6. DDL — Criando e Alterando Estruturas

Os comandos DDL definem o esquema do banco de dados — a estrutura que contém os dados. O mais fundamental é o CREATE TABLE, que cria uma nova tabela com suas colunas, tipos e constraints. Em PostgreSQL, cada coluna precisa de um tipo de dado que define o que pode ser armazenado nela: VARCHAR(n) para texto com limite, TEXT para texto sem limite, INTEGER e BIGINT para números inteiros, NUMERIC(p,s) para decimais precisos (use para dinheiro, nunca FLOAT), BOOLEAN para verdadeiro/falso, TIMESTAMP para datas e horas, e UUID para identificadores únicos.

O comando ALTER TABLE modifica a estrutura de uma tabela existente — adiciona ou remove colunas, altera tipos, adiciona ou remove constraints. Em produção, operações de ALTER em tabelas grandes podem travar a tabela por longos períodos (lock), por isso existem estratégias como adicionar a nova coluna como nullable primeiro, popular os dados em background e depois adicionar a constraint NOT NULL. Compreender o impacto de um ALTER é parte essencial do trabalho de um desenvolvedor de backend.

O comando DROP TABLE elimina uma tabela e todos os seus dados permanentemente. Em PostgreSQL, se a tabela tiver FKs referenciando ela de outras tabelas, você precisará usar DROP TABLE nome CASCADE (que também deleta as FKs das tabelas dependentes) ou primeiro remover as FKs manualmente. TRUNCATE é diferente: remove todas as linhas da tabela mas mantém a estrutura — e é muito mais rápido que DELETE sem WHERE porque não gera registros de undo linha a linha.

SQL — DDL: CREATE TABLE, ALTER TABLE, DROP TABLE 
-- Cria a tabela de usuários com tipos e constraints
CREATE TABLE usuarios (
  id         SERIAL       PRIMARY KEY,
  nome       VARCHAR(100) NOT NULL,
  email      VARCHAR(200) UNIQUE NOT NULL,
  ativo      BOOLEAN      DEFAULT TRUE,
  criado_em  TIMESTAMP    DEFAULT NOW()
);

-- Cria a tabela de pedidos referenciando usuarios
CREATE TABLE pedidos (
  id          SERIAL      PRIMARY KEY,
  cliente_id  INTEGER     NOT NULL REFERENCES usuarios(id),
  status      VARCHAR(20) NOT NULL DEFAULT 'pendente',
  criado_em   TIMESTAMP   DEFAULT NOW()
);

-- Adiciona coluna a tabela existente
ALTER TABLE usuarios ADD COLUMN telefone VARCHAR(20);

-- Remove tabela (irreversível!)
DROP TABLE IF EXISTS pedidos;

7. Constraints — Garantindo a Integridade dos Dados

As constraints (restrições) são regras que o SGBD aplica automaticamente a cada operação de escrita para garantir que os dados permaneçam válidos. Elas são a primeira linha de defesa contra dados inválidos — anteriores à validação na camada de aplicação. Se o SGBD rejeitar uma operação por violação de constraint, isso é um comportamento correto: é preferível um erro explícito a dados silenciosamente incorretos no banco.

As constraints mais comuns em PostgreSQL são: NOT NULL — a coluna não pode ter valor nulo; UNIQUE — nenhuma duplicata é permitida na coluna (ou combinação de colunas); DEFAULT — valor padrão caso nenhum seja fornecido na inserção; CHECK — expressão booleana que deve ser verdadeira para a linha ser aceita; PRIMARY KEY — combinação de NOT NULL e UNIQUE, define a PK da tabela; FOREIGN KEY — referência à PK de outra tabela, garante integridade referencial. Constraints podem ser definidas inline (junto à coluna) ou como constraints de tabela (ao final do CREATE TABLE), o que permite constraints sobre múltiplas colunas ao mesmo tempo.

Um detalhe importante: NULL é diferente de zero ou string vazia — NULL significa ausência de valor. Dois NULLs não são iguais entre si (NULL diferente de NULL em SQL). Isso tem implicações em consultas com WHERE coluna = NULL (que nunca retorna resultados — use WHERE coluna IS NULL) e no comportamento de constraints UNIQUE (em muitos SGBDs, múltiplos NULLs são permitidos mesmo com UNIQUE, porque NULL é considerado diferente de NULL). PostgreSQL segue esse comportamento padrão do SQL ANSI.

SQL — Constraints em ação 
CREATE TABLE produtos (
  id          SERIAL          PRIMARY KEY,
  nome        VARCHAR(150)     NOT NULL,
  sku         VARCHAR(50)      UNIQUE NOT NULL,
  preco       NUMERIC(10,2)    NOT NULL CHECK (preco > 0),
  estoque     INTEGER          NOT NULL DEFAULT 0
                                 CHECK (estoque >= 0),
  categoria   VARCHAR(50)      CHECK (categoria IN (
                                   'eletronico',
                                   'vestuario',
                                   'alimento'
                                 )),
  ativo       BOOLEAN          NOT NULL DEFAULT TRUE
);

-- Constraint de tabela: PK composta (tabela associativa N:M)
CREATE TABLE item_pedido (
  pedido_id   INTEGER NOT NULL REFERENCES pedidos(id) ON DELETE CASCADE,
  produto_id  INTEGER NOT NULL REFERENCES produtos(id),
  quantidade  INTEGER NOT NULL CHECK (quantidade > 0),
  preco_unit  NUMERIC(10,2) NOT NULL,
  PRIMARY KEY (pedido_id, produto_id)
);
Nunca use FLOAT para dinheiro

Tipos de ponto flutuante (FLOAT, DOUBLE PRECISION, REAL) têm representação binária que introduz erros de arredondamento. Para valores monetários, use sempre NUMERIC(precision, scale) — por exemplo NUMERIC(15,2) para valores em reais com centavos. Esse é um erro clássico que causa prejuízo real em sistemas financeiros.

8. DML — Manipulando Dados

Com a estrutura criada pelo DDL, passamos a manipular os dados com os comandos DML: INSERT, UPDATE e DELETE. Ao contrário dos comandos DDL, os comandos DML fazem parte de transações e podem ser revertidos com ROLLBACK se algo der errado. Isso é especialmente importante em operações que modificam múltiplas tabelas relacionadas, como uma transferência bancária ou um checkout de e-commerce.

O comando INSERT adiciona novas linhas a uma tabela. A forma básica insere uma linha por vez; a forma multi-linha insere várias linhas num único comando, o que é muito mais eficiente porque reduz o número de round-trips ao banco. PostgreSQL oferece a cláusula RETURNING, que retorna os valores das colunas inseridas — incluindo o id gerado pelo SERIAL — sem precisar de uma segunda consulta. Isso elimina a necessidade de um SELECT MAX(id) (que seria incorreto em ambiente concorrente) ou uma segunda busca após o INSERT.

Os comandos UPDATE e DELETE são poderosos e, portanto, perigosos sem a cláusula WHERE. Um UPDATE usuarios SET ativo = FALSE sem WHERE desativa todos os usuários. Um DELETE FROM pedidos sem WHERE apaga todos os pedidos permanentemente. Em PostgreSQL, você pode configurar a sessão para exigir WHERE em operações destrutivas, e muitas ferramentas como DBeaver e psql avisam antes de executar tais comandos. A regra prática: antes de executar qualquer UPDATE ou DELETE, primeiro rode um SELECT com o mesmo WHERE para verificar quantas e quais linhas serão afetadas.

SQL — DML: INSERT, UPDATE, DELETE 
-- INSERT básico (uma linha)
INSERT INTO usuarios (nome, email)
VALUES ('Ana Silva', 'ana@email.com');

-- INSERT múltiplo (mais eficiente)
INSERT INTO produtos (nome, sku, preco)
VALUES
  ('Camiseta Inteli', 'CAM-001', 59.90),
  ('Caderno A4',      'CAD-001', 24.90),
  ('Caneca Preta',    'CAN-001', 39.90);

-- INSERT com RETURNING (retorna o id gerado)
INSERT INTO pedidos (cliente_id, status)
VALUES (1, 'pendente')
RETURNING id, criado_em;

-- UPDATE com WHERE (sempre use WHERE!)
UPDATE usuarios
SET nome = 'Ana Costa', ativo = TRUE
WHERE id = 1;

-- DELETE com WHERE (sempre use WHERE!)
DELETE FROM usuarios
WHERE id = 1;
Nunca execute UPDATE ou DELETE sem WHERE em produção

Antes de qualquer UPDATE ou DELETE, rode primeiro o SELECT correspondente: SELECT * FROM usuarios WHERE id = 1. Confirme que as linhas retornadas são exatamente as que você quer modificar. Só então converta para UPDATE ou DELETE. Muitos acidentes em produção poderiam ser evitados com esse hábito simples.

9. DQL — Consultando Dados

O comando SELECT é, de longe, o mais utilizado em um banco de dados — aplicações leem dados muito mais frequentemente do que escrevem. O SELECT tem uma sintaxe rica e um motor de execução sofisticado que permite desde buscas simples até análises complexas com múltiplas agregações, filtros e ordenações. Dominar o SELECT é essencial para qualquer desenvolvedor backend.

A estrutura básica de um SELECT é: SELECT colunas FROM tabela WHERE condição ORDER BY coluna LIMIT n. O WHERE filtra linhas com base em condições booleanas — você pode combinar condições com AND, OR e NOT, usar operadores de comparação (=, <, >, <>), BETWEEN para intervalos, IN para listas de valores, LIKE para padrões de texto e IS NULL / IS NOT NULL para valores ausentes. O ORDER BY ordena o resultado por uma ou mais colunas em ordem ascendente (ASC, padrão) ou descendente (DESC). O LIMIT restringe o número de linhas retornadas — crucial para paginação e para evitar retornar milhões de linhas acidentalmente em tabelas grandes.

As funções de agregação calculam um único valor a partir de múltiplas linhas: COUNT(*) conta o total de linhas, COUNT(coluna) conta linhas onde a coluna não é NULL, SUM soma valores numéricos, AVG calcula a média aritmética, MIN e MAX retornam o menor e o maior valor. Funções de agregação normalmente são usadas junto com GROUP BY, que agrupa as linhas por uma ou mais colunas antes de aplicar a função. O HAVING filtra grupos (assim como WHERE filtra linhas individuais — HAVING só pode ser usado com GROUP BY).

SQL — DQL: SELECT com filtros, ordenação e agregações 
-- SELECT básico com WHERE, ORDER BY e LIMIT
SELECT nome, email
FROM usuarios
WHERE ativo = TRUE
  AND criado_em > '2024-01-01'
ORDER BY nome ASC
LIMIT 10;

-- DISTINCT: remove duplicatas no resultado
SELECT DISTINCT status
FROM pedidos;

-- Funções de agregação
SELECT
  COUNT(*) AS total_pedidos,
  SUM(preco_unit * quantidade) AS receita_total,
  AVG(preco_unit) AS ticket_medio,
  MIN(criado_em) AS primeiro_pedido,
  MAX(criado_em) AS ultimo_pedido
FROM pedidos
WHERE status = 'confirmado';

-- GROUP BY com HAVING: agrupa e filtra grupos
SELECT status, COUNT(*) AS qtd
FROM pedidos
GROUP BY status
HAVING COUNT(*) > 5
ORDER BY qtd DESC;

Aprofundamento: LIKE, ILIKE e padrões de texto

▼

O operador LIKE realiza busca por padrão em strings. O caractere % representa qualquer sequência de caracteres (incluindo nenhum), e _ representa exatamente um caractere. Exemplos:

  • WHERE nome LIKE 'Ana%' — começa com "Ana"
  • WHERE email LIKE '%@inteli.edu.br' — termina com o domínio
  • WHERE nome LIKE '%Silva%' — contém "Silva" em qualquer posição
  • WHERE nome LIKE 'A__' — começa com A e tem exatamente 3 caracteres

Para busca case-insensitive, use ILIKE (específico do PostgreSQL). Para buscas mais sofisticadas de texto completo (full-text search), PostgreSQL possui o operador @@ com vetores tsvector e queries tsquery — muito mais poderoso que LIKE para conteúdo extenso.

LIMIT e paginação

Para paginação, combine LIMIT e OFFSET: LIMIT 20 OFFSET 40 retorna as linhas 41 a 60 (página 3 com 20 itens por página). Mas em tabelas muito grandes, OFFSET alto é lento — o banco ainda precisa processar e descartar as primeiras N linhas. Para paginação de alta performance, prefira a técnica de "keyset pagination" (cursor pagination) usando a PK como referência da última linha vista.

10. DCL e TCL — Controle de Acesso e Transações

Os comandos DCL controlam quem pode fazer o quê no banco de dados. GRANT concede privilégios a usuários ou roles; REVOKE remove privilégios. Os privilégios podem ser granulares: SELECT, INSERT, UPDATE, DELETE, REFERENCES (para criar FKs). O princípio do menor privilégio diz que cada usuário ou aplicação deve ter apenas os privilégios mínimos necessários para sua função. Uma API que só lê dados deve ter apenas SELECT; uma API que escreve deve ter SELECT, INSERT, UPDATE, DELETE nas tabelas relevantes — mas nunca DROP TABLE ou acesso ao schema de sistema.

Os comandos TCL controlam transações. Uma transação é um grupo de operações que deve ser executado de forma atômica: ou todas têm sucesso, ou nenhuma é persistida. BEGIN inicia a transação; COMMIT persiste todas as mudanças; ROLLBACK desfaz tudo desde o BEGIN. O exemplo clássico é a transferência bancária: você precisa debitar de uma conta E creditar em outra — se o crédito falhar após o débito ter sido realizado, o dinheiro simplesmente desapareceria. A transação garante que isso não aconteça.

As transações em bancos relacionais seguem as propriedades ACID: Atomicidade — a transação é indivisível (tudo ou nada); Consistência — a transação leva o banco de um estado válido a outro estado válido, respeitando todas as constraints; Isolamento — transações concorrentes não interferem entre si (o nível de isolamento pode ser configurado); Durabilidade — após o COMMIT, os dados são persistidos mesmo em caso de falha de hardware (o SGBD escreve em um WAL — Write-Ahead Log — antes de qualquer outra coisa).

SQL — DCL: GRANT/REVOKE e TCL: BEGIN/COMMIT/ROLLBACK 
-- DCL: conceder e revogar privilégios
GRANT SELECT ON usuarios TO leitor;
GRANT SELECT, INSERT, UPDATE, DELETE ON pedidos TO api_app;
REVOKE DELETE ON usuarios FROM api_app;

-- TCL: transferência bancária atômica
BEGIN;

UPDATE contas
SET saldo = saldo - 100.00
WHERE id = 1 AND saldo >= 100.00;

UPDATE contas
SET saldo = saldo + 100.00
WHERE id = 2;

-- Se tudo correu bem, persiste as mudanças:
COMMIT;

-- Se algo deu errado, desfaz tudo:
-- ROLLBACK;

12. RM-ODP — Viewpoint Information

O RM-ODP (Reference Model for Open Distributed Processing, ISO/IEC 10746) estrutura a descrição de sistemas distribuídos em cinco viewpoints (visões). Cada visão captura os interesses de um grupo específico de stakeholders, usando uma linguagem e nível de abstração adequados a esses interesses. As cinco visões são: Enterprise (propósito e regras de negócio), Information (estrutura e fluxo de dados), Computational (decomposição funcional em componentes), Engineering (distribuição e comunicação) e Technology (escolhas concretas de tecnologia).

Esta aula foca no Viewpoint Information — a visão de informação. Esse viewpoint descreve como a informação é estruturada, como muda ao longo do tempo e quais invariantes (regras permanentes) ela deve obedecer. No contexto de banco de dados, o viewpoint de informação se traduz diretamente no modelo ER e no esquema relacional: as entidades são os objetos de informação, os atributos são suas propriedades, os relacionamentos são as associações entre eles, e as constraints são os invariantes que o sistema deve preservar em toda operação.

Quando você escreve um diagrama ER, você está produzindo a visão de informação do sistema. Quando você define as constraints na DDL (NOT NULL, UNIQUE, CHECK, FOREIGN KEY), você está codificando os invariantes de informação diretamente no SGBD. Isso é poderoso: as regras de negócio relacionadas à estrutura de dados ficam centralizadas no banco, não espalhadas pela lógica da aplicação onde podem ser esquecidas ou inconsistentes entre diferentes endpoints da API.

13. RF, RN e RNF aplicados a Banco de Dados

Toda funcionalidade que precisa persistir dados possui Requisitos Funcionais (RF) que dependem do banco de dados. Identificar essa dependência é o primeiro passo para modelar corretamente. Um RF como "o usuário deve poder visualizar seu histórico de pedidos" implica que os pedidos precisam ser armazenados com referência ao usuário, com data, status e itens — e que a consulta deve ser eficiente o suficiente para retornar em tempo aceitável mesmo com milhares de pedidos por usuário. Esse RF, ao ser implementado, exige decisões de modelagem (quais tabelas, quais colunas), de DDL (tipos e constraints) e de DQL (qual SELECT e quais índices).

As Regras de Negócio (RN) que envolvem dados frequentemente se traduzem em constraints no banco. "Um produto não pode ter estoque negativo" vira CHECK (estoque >= 0). "Um email de usuário deve ser único no sistema" vira UNIQUE NOT NULL. "Um pedido só pode ter status 'confirmado', 'pendente' ou 'cancelado'" vira um CHECK (status IN (...)). Quando a RN é simples o suficiente para ser expressa como constraint SQL, prefira o banco à aplicação — o banco a aplicará em todo acesso, de qualquer origem, incluindo scripts de manutenção e migrações.

Os Requisitos Não Funcionais (RNF) mais diretamente impactados pelo banco de dados são os de Confiabilidade, Desempenho, Segurança e Suportabilidade. A escolha de usar transações ACID (TCL) é uma decisão de RNF de Confiabilidade. A criação de índices para acelerar consultas frequentes é um RNF de Desempenho. O uso de roles com menor privilégio (DCL) é um RNF de Segurança. A estruturação das migrações de banco com ferramentas como Flyway ou Knex é um RNF de Suportabilidade que garante que o esquema possa evoluir de forma controlada e rastreável.

14. Prática Hands-on — Crie sua primeira tabela no Supabase

Toda a teoria deste material só vira competência real quando você executa SQL contra um banco de dados de verdade. Esta seção propõe uma prática curta (~10 min) usando o Supabase — uma plataforma cloud que oferece PostgreSQL gerenciado com SQL Editor embutido no navegador. Você vai modelar uma entidade pequena do projeto e ver, no Table Editor, a tabela aparecer com os tipos e constraints que escreveu.

Por que Supabase?

Supabase é PostgreSQL puro, sem dialeto próprio. Tudo que você aprende lá funciona em qualquer PostgreSQL self-hosted, RDS ou Cloud SQL. A interface elimina o atrito de instalação local — basta um login com GitHub/Google e você está dentro de um SQL Editor totalmente funcional, com autocomplete, execução por Ctrl+Enter e o resultado aparecendo lado a lado.

Passo a passo

  1. Acesse supabase.com e faça login (GitHub, Google ou e-mail).
  2. Crie um projeto novo ou abra o projeto da sua squad. Anote a senha do banco de dados — Supabase só mostra uma vez.
  3. Abra o SQL Editor no menu lateral (ícone </>).
  4. Identifique uma entidade pequena do problema que sua dupla está modelando — algo como cliente, agendamento, tarefa ou produto. Não precisa modelar o domínio inteiro.
  5. Modele 3 a 5 colunas: uma PK, dois ou três campos descritivos e ao menos uma constraint útil (UNIQUE, NOT NULL ou CHECK).
  6. Escreva o CREATE TABLE em PostgreSQL no editor.
  7. Execute com Ctrl+Enter (ou clique em Run). Você deve ver "Success. No rows returned." se tudo deu certo.
  8. Confirme em Table Editor (ícone de tabela no menu lateral) que a tabela apareceu com as colunas e tipos corretos.
  9. Bônus: insira uma linha com INSERT INTO ... e rode SELECT * FROM ... para ver o registro aparecer.
SQL — exemplo de referência (ajuste para a entidade do seu projeto) 
-- Exemplo: tabela de clientes para uma loja universitária
CREATE TABLE clientes (
  id         SERIAL       PRIMARY KEY,
  nome       VARCHAR(120) NOT NULL,
  email      VARCHAR(200) UNIQUE NOT NULL,
  curso      VARCHAR(50)  CHECK (curso IN ('CC', 'EC', 'SI', 'ES', 'AT')),
  criado_em  TIMESTAMP    DEFAULT NOW()
);

-- Bônus: insere uma linha e consulta o resultado
INSERT INTO clientes (nome, email, curso)
VALUES ('Ana Lima', 'ana@inteli.edu.br', 'SI')
RETURNING id, nome, criado_em;

SELECT * FROM clientes;

Critério de sucesso

  • ✅ A tabela aparece no Table Editor com as colunas e tipos esperados.
  • ✅ A constraint UNIQUE bloqueia uma segunda inserção com o mesmo email (teste e veja o erro do PostgreSQL).
  • ✅ A constraint CHECK rejeita um valor de curso fora da lista permitida.
  • ✅ Você consegue explicar para sua dupla, em uma frase, por que cada constraint foi escolhida.
Erros mais comuns

Senha do banco esquecida: dá para resetar em Project Settings → Database, mas isso desconecta clientes existentes — fique atento.
Tabela duplicada: use CREATE TABLE IF NOT EXISTS ou rode DROP TABLE clientes; antes do CREATE para reexecutar o script sem erro.
Schema "public": Supabase coloca tudo em public por padrão — não é problema, mas em produção real você organizaria por schema (auth, billing, etc.).
