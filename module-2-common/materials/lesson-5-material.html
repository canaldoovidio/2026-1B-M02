1. De onde paramos · onde vamos

Na aula 4 vocês construíram um sandbox SQLite no navegador: sql.js via WebAssembly + dataset BrPec carregado por fetch. Funciona offline (depois do primeiro load) e responde queries em milissegundos.

Mas tem três limitações que impedem a PWA do BrPec de ser útil em produção:

  1. Não persiste mudanças. Um INSERT funciona — até o reload. O motor mora em memória; o blob carregado por fetch sempre volta ao seed original.
  2. Não é instalável. É uma aba do Chrome. O capataz, quando precisar registrar uma pesagem no Cocho Novo, vai ter que abrir o navegador, digitar URL, navegar… ou esquecer e usar papel.
  3. Não fala com servidor. Cada celular tem seu sandbox isolado. O coordenador não tem dashboard consolidado. Sem caminho de volta para o SGBD da fazenda.

Esta aula resolve as três limitações com técnicas precisas — mas, antes da PWA, vamos olhar com cuidado para o back-end que recebe os dados quando o sinal volta:

  • O que é Node.js e por que ele encaixa bem nesse ecossistema (mesma linguagem da PWA).
  • Como organizar o servidor em Models e Controllers (SWEBOK v4 Cap. 2) — separar estrutura e comportamento.
  • Como conectar esse servidor ao SQLite da aula 4 (mesmo motor, do outro lado da rede) via better-sqlite3.
  • Como sustentar esse código com TDD bem feito — teste de porta, não de entranha — e abrir janela para Spec-Driven Development.
  • E só então: persistência (IndexedDB), instalabilidade (manifest + SW) e sync idempotente (INSERT OR IGNORE + client_uuid) — revisita compacta para amarrar o ciclo.

2. O que é Node.js?

Node.js é um runtime que executa JavaScript fora do navegador. Usa o motor V8 (o mesmo do Chrome) + APIs de sistema (arquivos, rede, processos). Com isso, o mesmo JS que rodava só no navegador agora roda em servidores, ferramentas de linha de comando, e tudo entre os dois.

3 ideias-âncora

  • Event loop. Um único thread principal cuida de I/O assíncrono (rede, disco). Enquanto uma requisição espera o banco, o servidor já atende outras 1000. Perfeito para servidores web — muita conexão, pouca CPU por requisição.
  • npm. O maior ecossistema de pacotes do mundo. express (HTTP), better-sqlite3 (SQLite), supertest (testar HTTP), cors — todos viram npm install <pkg> e require('<pkg>').
  • Mesma linguagem dos dois lados. A PWA da aula 4 é JS. O servidor de hoje é JS. Modelos e validações podem ser literalmente compartilhados via módulos comuns. Sem context switch entre linguagens.

Server HTTP em 8 linhas

// servidor.js
const express = require('express');
const app = express();
app.use(express.json());
app.get('/ping', (req, res) => {
  res.json({ ok: true, ts: Date.now() });

app.listen(3000, () => console.log('BrPec rodando em :3000'));

3. SWEBOK v4 · Cap. 2 · base teórica de Models e Controllers

Antes de "MVC vira jargão", vale ancorar a ideia na literatura. O SWEBOK (Software Engineering Body of Knowledge) v4, da IEEE Computer Society (2024), dedica o Capítulo 2 do Knowledge Area "Software Design" para o que chama de Structural and Behavioral Design.

Os dois lados do design

  • Structural design — como o sistema é dividido em partes: módulos, camadas, interfaces, dependências. Resposta a "quem faz o quê?"
  • Behavioral design — como as partes colaboram em tempo de execução: fluxos de dados, sequências de eventos, casos de uso. Resposta a "como elas conversam?"

A regra-ouro: defina os limites antes dos fluxos. Cada parte deve ter um motivo para mudar (Single Responsibility Principle).

Por que isso importa agora?

Quando você escreve um servidor inteiro em um único app.js com rotas, SQL, validações e regras de negócio todos misturados, ele funciona — até começar a crescer. Daí o primeiro bug pequeno vira refactor gigante porque todo mundo conhece todo mundo.

MVC é uma escolha estrutural (3 camadas, 3 responsabilidades). Os controllers definem o comportamento (o que cada rota faz). Mantendo as camadas separadas:

  • Trocar o banco (SQLite → Postgres) toca apenas o Model.
  • Trocar o transporte (HTTP → fila de mensagens) toca apenas o Routes + Controller.
  • Testar regra de negócio pura: chama o Model direto, sem subir HTTP.
Trio para construir intuição teórica

① SWEBOK v4 Cap. 2 — vocabulário canônico. ② Livro Didático — Teste de Unidade (Ovidio) — como testar cada parte isoladamente. ③ Test-Driven Development de Kent Beck (caps. 1–3) — o ritmo que faz a estrutura emergir naturalmente.

4. MVC aplicado ao BrPec

"Model, View, Controller" é uma sigla antiga (1979 · Trygve Reenskaug · Smalltalk-80). No contexto de servidores HTTP modernos, ela se ajusta assim:

CamadaResponde aNo BrPec hoje
ModelComo o estado é guardado? Qual a regra de negócio? Qual a invariante?evento.model.js — schema da tabela evento_recebido + statements preparados + funções insertMany, findRecent.
ViewComo o estado vira saída para o cliente?No HTTP/JSON: a View é o objeto retornado por res.json(...) no Controller.
ControllerQuem orquestra: recebe request, valida, chama o model, monta resposta?sync.controller.js — postDeltas(req, res), getSnapshot(req, res).
RoutesQue URL chama qual controller?server.js — app.post('/sync/deltas', postDeltas).

As 3 regras que não se quebra

  1. Model nunca conhece req/res. Se conhecer, quando você trocar de HTTP para fila (Kafka, RabbitMQ), tudo quebra. Model fala SQL e regras, e só.
  2. Controller nunca escreve SQL. Se escrever, quando você trocar SQLite por Postgres (ou MongoDB), tudo quebra. Controller orquestra, valida, formata.
  3. Routes não tem lógica. Só liga URL → função do Controller. Se for tentado a adicionar if em route, é sinal de que o controller precisa de uma sub-rotina.

Estrutura de pastas que vamos seguir

module-2-common/assets/brpec-sync-server/ server.js # ROUTES + bootstrap do Express controllers/ sync.controller.js # CONTROLLER · valida payload, chama Model models/ evento.model.js # MODEL · SQL + statements + transações server.test.js # testes contra a PORTA HTTP package.json
Conexão com a aula 4

O Model do BrPec usa exatamente as mesmas tabelas e os mesmos JOINs que vocês aprenderam ontem (capataz, retiro, bovino, evento_zootecnico). A diferença é só: agora o motor SQLite roda no servidor Node, não no navegador. O SQL é o mesmo; muda quem digita.

5. PWA · três pilares

Uma Progressive Web App é uma página web que ganha 3 superpoderes: instalabilidade, capacidade offline e acesso a APIs nativas (notificações, sync em background, câmera, …). Os superpoderes vêm de três tecnologias mínimas:

🔒 HTTPS (ou localhost)

Sem HTTPS, o navegador bloqueia Service Worker, crypto.subtle, e várias outras APIs. Em produção é obrigatório — GitHub Pages serve via HTTPS. Em dev, localhost é exceção válida.

📄 Manifest (.webmanifest)

JSON declarativo com nome, ícones, cor de tema, modo de exibição. É o que o navegador lê para oferecer "Instalar" e para configurar a janela quando o app abre.

⚙️ Service Worker

Script JavaScript que vive fora da página (em uma thread própria, com ciclo de vida próprio). Intercepta requisições, cacheia recursos, responde quando offline. É o coração do offline-first.

Lighthouse PWA score

A ferramenta automática do Chrome avalia exatamente esses 3 pilares + alguns detalhes (ícone ≥ 192px, start_url cacheável, etc). Score alto = checklist completo.

7. Service Worker · ciclo de vida

O SW tem 3 eventos importantes. Dois você implementa, um o navegador dispara para cada requisição.

install · pre-cache da shell

Roda uma vez por versão do SW. O lugar para carregar o cache com tudo que o app precisa para subir offline: HTML, CSS, JS, manifest, ícone, motor WASM, banco seed.

activate · limpeza de caches antigos

Roda quando o SW vira "current". Aqui você deleta caches de versões antigas. self.clients.claim() faz o SW assumir tabs já abertas; self.skipWaiting() no install evita ficar em "waiting".

fetch · interceptação de requisições

Toda requisição saindo da página passa por aqui. É onde você implementa a estratégia (cache-first, network-first, stale-while-revalidate, …).

// module-2-common/assets/brpec-pwa/sw.js
const CACHE_NAME = 'brpec-pwa-v1';
const SHELL_ASSETS = [
  './', './index.html', './app.js', './db.js',
  './styles.css', './manifest.webmanifest', './icon.svg',
  '../brpec/brpec.db',
  'https://cdn.jsdelivr.net/npm/sql.js@1.13.0/dist/sql-wasm.js',
  'https://cdn.jsdelivr.net/npm/sql.js@1.13.0/dist/sql-wasm.wasm'
];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => cache.addAll(SHELL_ASSETS))
  );
  self.skipWaiting();
});

self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(keys.filter((k) => k !== CACHE_NAME).map((k) => caches.delete(k)))
    )
  );
  self.clients.claim();
});
Dica de versionamento

Versione o nome do cache (brpec-pwa-v1 → v2) toda vez que mudar a lista de assets ou a lógica do SW. Isso força o navegador a baixar o novo SW e descartar o antigo.

8. Estratégias de cache · qual usar onde?

A regra básica: shell estática usa cache-first; dados frescos usam network-first.

cache-first · para a shell

Tenta o cache. Se achou, devolve direto. Senão, busca rede e cacheia. Bom para HTML, CSS, JS, WASM, ícones, banco seed — coisas que mudam só com deploy.

caches.match(event.request).then((cached) => {
  if (cached) return cached;
  return fetch(event.request).then((resp) => {
    if (resp.ok) {
      const clone = resp.clone();
      caches.open(CACHE_NAME).then((c) => c.put(event.request, clone));
    }
    return resp;
  });
});

network-first · para dados

Tenta a rede primeiro. Se falhou, devolve fallback estruturado (não a página do erro do navegador). Bom para /sync/* — você quer fresh, mas degrada graciosamente.

fetch(event.request).catch(() =>
  new Response(
    JSON.stringify({ ok: false, reason: 'offline' }),
    { status: 503, headers: { 'content-type': 'application/json' } }
  )
);
No SW do BrPec

O sw.js implementa exatamente isso: cache-first para a shell, network-first com fallback 503 para qualquer /sync/*. Releia o arquivo — são 60 linhas didáticas.

9. Persistência · sql.js + IndexedDB

O sandbox da aula 4 carrega o banco em memória — fim do reload, banco volta ao seed. Para persistir, gravamos o blob binário do banco no IndexedDB depois de cada mutação. No próximo boot, lemos de lá em vez do seed.

Fluxo de boot (3 passos)

  1. Motor — initSqlJs() carrega o WASM (cacheado pelo SW).
  2. Blob — tenta ler do IndexedDB (key brpec.db). Se vazio, fetch o seed inicial e grava no IDB.
  3. Banco — new SQL.Database(blob) abre em memória, pronto pra usar.
// db.js — núcleo da persistência
export async function initDb() {
  const SQL = await initSqlJs({
    locateFile: (f) => `https://cdn.jsdelivr.net/npm/sql.js@1.13.0/dist/${f}`
  });

  let blob = await idbGet('brpec.db');
  if (!blob) {
    blob = await fetchSeed();   // busca brpec.db da aula 4
    await idbSet('brpec.db', blob);
  }

  const db = new SQL.Database(blob);
  db.run(`CREATE TABLE IF NOT EXISTS evento_zootecnico_pending ( ... );`);

  async function persist() {
    await idbSet('brpec.db', db.export());  // blob → IDB
  }

  return { db, persist };
}

Depois de cada INSERT/UPDATE/DELETE, chame await persist(). Isso serializa o estado atual e grava no IDB. Próximo boot, reabre exatamente onde parou.

Por que não OPFS?

O Origin Private File System (OPFS) é mais eficiente que IndexedDB para arquivos grandes — mas exige crossOriginIsolated, que precisa de headers COOP/COEP. GitHub Pages não envia esses headers. Trade-off: IndexedDB é universalmente suportado e funciona em qualquer host; aceitamos um overhead pequeno (gravar o blob inteiro a cada mutação) em troca de portabilidade.

Para um banco com ~30 bovinos e algumas dezenas de eventos (BrPec MVP), isso é < 50 KB — gravar em IDB leva < 10 ms. Imperceptível.

10. UUID v4 · gerado no celular do capataz

Cada evento, no momento da coleta, recebe um client_uuid gerado localmente. Esse id acompanha o evento toda vez que ele atravessa a rede — e é a base da idempotência.

Por que não deixar o servidor gerar o id?

  • O servidor pode estar inacessível no momento da coleta.
  • O cliente precisa identificar localmente o evento (ex.: para mostrar "registrado em 14:32") sem esperar o servidor confirmar.
  • Quando o sync acontece e a rede cai no meio, o cliente reenvia — e o servidor precisa reconhecer "isso já chegou".

Geração portátil

crypto.randomUUID() só funciona em contextos seguros (HTTPS ou localhost). Para a sala de aula, onde a PWA pode ser servida via IP de LAN, precisamos de fallback:

function makeUUID() {
  if (window.crypto && typeof crypto.randomUUID === 'function') {
    try { return crypto.randomUUID(); } catch (_) { /* fall through */ }
  }
  // Fallback: getRandomValues funciona em qualquer contexto
  const b = new Uint8Array(16);
  crypto.getRandomValues(b);
  b[6] = (b[6] & 0x0f) | 0x40;  // version 4
  b[8] = (b[8] & 0x3f) | 0x80;  // variant 10
  const h = [...b].map((x) => x.toString(16).padStart(2, '0'));
  return `${h[0]}${h[1]}${h[2]}${h[3]}-${h[4]}${h[5]}-${h[6]}${h[7]}-${h[8]}${h[9]}-${h.slice(10).join('')}`;
}
Sobre colisão

UUID v4 tem 122 bits aleatórios — 2¹²² combinações. Para gerar uma colisão com 50% de probabilidade, você precisaria gerar ~2,7 × 10¹⁸ UUIDs. Não é problema na prática desde 2005.

11. Estratégia de sync · arquitetura do reencontro

Cliente acumula offline. Servidor reconcilia idempotentemente. Sem locks, sem janela de conflito.

Push de deltas (foco do MVP)

Fluxo: cliente lista eventos com synced=0 → monta payload → POST /sync/deltas → servidor responde com {accepted, skipped, accepted_uuids} → cliente marca como synced=1.

// app.js — função de sync
async function onSincronizar() {
  const pending = listToSync(db);   // SELECT ... WHERE synced = 0
  if (!pending.length) return;

  const resp = await fetch(`${SERVER_URL}/sync/deltas`, {
    method: 'POST',
    headers: { 'content-type': 'application/json' },
    body: JSON.stringify({ deltas: pending })
  });
  if (!resp.ok) throw new Error(`HTTP ${resp.status}`);

  const data = await resp.json();
  markSynced(db, data.accepted_uuids);  // UPDATE local
  await persist();                       // blob → IDB
}

Pull de snapshot (futuro)

Cliente pode pedir GET /sync/snapshot para conferir o que o servidor sabe. Útil em cenário "voltei a ter rede e quero saber o que perdi". Para o MVP da aula 5, o foco é push.

Conflitos verdadeiros — fora de escopo

"Dois capatazes editaram o mesmo bovino offline". A solução varia muito (last-write-wins, merge manual, OT/CRDT…). Em uma aula introdutória de offline-first, ficamos com o caso comum: cada coleta é um evento imutável com id único — não há "edição concorrente do mesmo registro".

12. Servidor Node · MVC com Express + better-sqlite3

O servidor é deliberadamente minúsculo (~120 linhas com comentários generosos), mas organizado em MVC: cada arquivo tem uma e apenas uma responsabilidade. Releia a seção 4 se precisar refrescar o porquê.

Anatomia (versão didática · MVC visível)

const express  = require('express');
const cors     = require('cors');
const Database = require('better-sqlite3');

function createServer({ dbPath = './brpec-server.db' } = {}) {
  const db = new Database(dbPath);
  db.pragma('journal_mode = WAL');  // concorrência leitura/escrita
  db.exec(`
    CREATE TABLE IF NOT EXISTS evento_recebido (
      client_uuid  TEXT PRIMARY KEY,         -- a mágica
      tipo         TEXT NOT NULL,
      data         TEXT NOT NULL,
      capataz_id   INTEGER NOT NULL,
      retiro_id    INTEGER NOT NULL,
      bovino_id    INTEGER NOT NULL,
      observacao   TEXT,
      received_at  TEXT NOT NULL DEFAULT (datetime('now'))
    );
  `);

  const stmt = db.prepare(`
    INSERT OR IGNORE INTO evento_recebido
      (client_uuid, tipo, data, capataz_id, retiro_id, bovino_id, observacao)
    VALUES
      (@client_uuid, @tipo, @data, @capataz_id, @retiro_id, @bovino_id, @observacao);
  `);

  const app = express();
  app.use(cors());
  app.use(express.json({ limit: '512kb' }));

  app.post('/sync/deltas', (req, res) => {
    const deltas = req.body?.deltas;
    if (!Array.isArray(deltas)) return res.status(400).json({ ok: false });

    const tx = db.transaction((items) => {
      const accepted_uuids = []; let skipped = 0;
      for (const d of items) {
        if (!d.client_uuid || !d.tipo || ...) { skipped++; continue; }
        const info = stmt.run(d);
        if (info.changes === 1) accepted_uuids.push(d.client_uuid);
        else                    skipped++;
      }
      return { accepted: accepted_uuids.length, skipped, accepted_uuids };
    });

    res.json({ ok: true, ...tx(deltas) });
  });

  return { app, db, close: () => db.close() };
}

13. TDD · Red → Green → Refactor + spec mínimo

Test-Driven Development não é "escrever testes". É um ritmo de trabalho em 3 tempos:

  • 🔴 RED — escrever um teste que descreve o comportamento desejado. Ele falha (não existe código para passar). Esse "vermelho" é o contrato.
  • 🟢 GREEN — escrever o mínimo de código de produção para passar. Não tente resolver problemas futuros. Só passa o teste.
  • 🔵 REFACTOR — agora que está verde, reorganize: extrair função, renomear, eliminar duplicação. Os testes verdes formam a rede de segurança.

Mantemos a cultura TDD com 1 teste-âncora: idempotência via supertest. O contrato-chave da aula é justamente esse — testá-lo é forçar que o servidor exista para cumpri-lo.

Por que esse teste em particular?

Idempotência é o contrato pedagógico da aula 5: o que separa "sync ingênuo" de "sync confiável". E é o teste mais ilustrativo de uma boa prática: atacar a porta do sistema, não as entranhas.

Observe: chamamos POST /sync/deltas via supertest — não insertMany() direto do Model. Isso é deliberado. Se amanhã quebrarmos o Model em duas funções (validate() + insert()), o teste continua verde. O teste descreve o contrato HTTP, não o detalhe da implementação.

// server.test.js
const test   = require('node:test');
const assert = require('node:assert/strict');
const request = require('supertest');
const { createServer } = require('./server');

test('reenviar não duplica (idempotência via client_uuid)', async () => {
  const { app, close } = createServer({ dbPath: ':memory:' });

  const delta = {
    client_uuid: 'fixed-uuid-A',
    tipo:        'pesagem',
    data:        '2026-05-14',
    capataz_id: 1, retiro_id: 1, bovino_id: 1,
    observacao: '500kg'
  };

  const r1 = await request(app).post('/sync/deltas').send({ deltas: [delta] });
  assert.equal(r1.body.accepted, 1);
  assert.equal(r1.body.skipped,  0);

  const r2 = await request(app).post('/sync/deltas').send({ deltas: [delta] });
  assert.equal(r2.body.accepted, 0);
  assert.equal(r2.body.skipped,  1);

  close();
});

14. TDD bem feito · "Where Did It All Go Wrong" + Spec-Driven Development

Duas leituras provocadoras para fechar o bloco de qualidade da aula. A primeira mostra por que tantas equipes adotam TDD e não veem resultado. A segunda aponta para onde a indústria está indo depois.

🔧 TDD, Where Did It All Go Wrong (Ian Cooper · DevTernity 2017)

Tese central: a maioria das equipes faz TDD errado — testa implementação em vez de comportamento. Resultado: suíte de testes frágil, qualquer refactor quebra 30 testes, equipe começa a odiar testes, abandona TDD.

O que mudar: escrever testes contra as portas (pontos de entrada do sistema — endpoints HTTP, comandos CLI, callbacks de eventos), não contra as entranhas (classes/funções internas). Cada bug encontrado vira 1 teste novo, mas só na borda.

Aplicado ao BrPec: nosso teste de idempotência ataca a porta HTTP (POST /sync/deltas + body + status code). Se amanhã quebrarmos insertMany em validate + insertOne, o teste continua verde. Esse é o ritmo certo: refactor livre internamente, garantia constante na borda.

📐 Understanding Spec-Driven Development (Kiro · spec-kit · Tessl)

Tese central: a próxima fronteira não é "escrever testes antes do código". É declarar a especificação e deixar a IA gerar, em conjunto, (1) o código de produção, (2) os testes que validam essa spec, (3) documentação coerente.

3 ferramentas para conhecer:

FerramentaOrigemPosição
KiroAWS · 2024IDE com "spec-mode" embutido — escreve .spec.md, IA gera + sincroniza implementação.
spec-kitGitHub · 2025CLI open-source. Roda em qualquer projeto, qualquer linguagem. Gera código a partir de specs versionadas.
TesslTessl AI · 2025Plataforma SaaS — specs executáveis como artefato primário; "código" é derivado.

No BrPec hoje: nosso "spec" ainda é texto em prosa + 1 teste-âncora. Em ~2 anos, esperamos que brpec.spec.md gere server.js + server.test.js automaticamente. A direção é clara; só falta amadurecimento das ferramentas.

A mensagem para a turma 26

Testes não são burocracia. São o contrato executável que sobrevive a refactors, mudanças de stack, e até a substituições do autor original do código. Quem domina TDD bem feito (Ian Cooper) ganha velocidade. Quem se prepara para SDD ganha relevância nos próximos 5 anos.
