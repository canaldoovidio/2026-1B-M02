1. Por que testar

Existe um momento na vida de todo projeto em que ele cresce o suficiente para que ninguém — nem mesmo quem o escreveu — consiga prever todas as consequências de uma alteração. É o ponto em que mexer no código vira loteria: a feature nova funciona, mas quebra um cadastro antigo; o refactor "óbvio" derruba um endpoint que ninguém lembrava que existia; um bugzinho bobo passa pela revisão e só aparece em produção, na sexta-feira, depois das 18h. Testes automatizados existem para que esse momento nunca chegue.

Há quatro razões concretas para investir em testes desde cedo, mesmo (ou principalmente) num projeto pequeno. A primeira é regressão: cada teste verde é uma trava contra um bug que você já caçou e não quer caçar de novo. A segunda é refactor seguro: com cobertura razoável, reorganizar pastas, renomear classes ou trocar uma biblioteca deixa de ser um ato de fé. A terceira é documentação executável: ler describe('UserService.create') com seis casos descritos é melhor do que qualquer README sobre como o serviço se comporta. A quarta é confiança no deploy: se o pipeline de CI passou — lint, types, testes — você pode mergear na sexta-feira sem suar frio.

A premissa desta aula

Vamos assumir o stack do projeto: TypeScript + Express + EJS + PostgreSQL em arquitetura MVC de seis camadas (models, views, controllers, services, repositories, helpers). Cada camada tem uma estratégia de teste que combina com sua responsabilidade — e é justamente combinar bem cada estratégia que diferencia uma suíte útil de uma suíte que ninguém olha.

Quando o ROI vira negativo

Testes não são gratuitos. Eles custam tempo para escrever, tempo para rodar e tempo para manter. A pergunta certa não é "preciso de testes?" — é "onde a relação custo/benefício compensa?". Em geral: lógica de negócio em services merece muitos testes; helpers puros são baratíssimos de testar; controllers ganham com integração; views EJS quase nunca precisam de teste unitário. Vamos justificar cada escolha ao longo do material.

2. Pirâmide de testes

Mike Cohn popularizou em 2009 a metáfora da pirâmide de testes: muitos testes unitários na base, alguns testes de integração no meio e poucos testes end-to-end no topo. A justificativa é mecânica: testes unitários são rápidos (milissegundos), determinísticos e baratos de manter; testes E2E são lentos (segundos a minutos), mais frágeis (dependem de rede, banco, navegador) e caros para investigar quando falham. Inverter essa proporção é desperdiçar tempo de CI e energia de gente.

E2Epoucos · lentos · alta confiança
Integraçãomédios · cobre fronteiras
Unitáriosmuitos · rápidos · isolados

Cada nível responde a uma pergunta diferente. Unitário: "essa função pura, com essas entradas, retorna isso?". Integração: "esses módulos conversam corretamente — controller fala com service, service fala com repo?". E2E: "quando um usuário real abre o navegador e preenche o formulário, o cadastro acontece e o e-mail chega?". Os três tipos coexistem; nenhum substitui outro.

Anti-padrões: ice-cream cone e hourglass

O ice-cream cone é uma pirâmide invertida: muito E2E, pouquíssimo unitário. Acontece em times que começam testando "do jeito do usuário" e nunca descem o nível. O resultado é uma suíte que demora 40 minutos para rodar, falha por motivos aleatórios (a rede do CI, um seletor de CSS que mudou) e ninguém confia. O hourglass tem muitos unitários e muitos E2E, mas pouca integração no meio: o problema é que bugs reais costumam acontecer na fronteira entre módulos — e essa fronteira fica sem cobertura.

NívelO que cobreVelocidade típicaQuando falha
Unitário1 função/classe isolada< 50msLógica errada na unidade
Integração2+ camadas conversando50ms - 2sContrato entre camadas quebrou
E2EFluxo completo do usuário5s - 60sCaminho real do usuário quebrou
Proporção razoável para um projeto MVC

Uma proporção saudável para o nosso stack é algo como 70% unitários (helpers + services com mocks), 25% integração (repositories e controllers com supertest) e 5% E2E (Playwright apenas em fluxos críticos: login, cadastro, checkout). Isso é uma sugestão, não uma lei — calibre conforme o domínio.

3. Estrutura AAA e naming

Todo bom teste tem três partes claras: Arrange (preparar o cenário), Act (executar a ação) e Assert (verificar o resultado). É o padrão AAA. Separá-los com linhas em branco — ou comentários nas primeiras semanas — força quem lê a entender em segundos o que está sendo testado.

TypeScript — estrutura AAA
import { formatCurrency } from '../helpers/format';

describe('formatCurrency', () => {
  it('formata número positivo em BRL', () => {
    // Arrange
    const valor = 1234.5;

    // Act
    const saida = formatCurrency(valor);

    // Assert
    expect(saida).toBe('R$ 1.234,50');
  });
});

Sobre naming: a regra prática é que ao ler apenas describe + it, sem o corpo, deveria dar para entender o comportamento testado. Bons exemplos: describe('UserService.create') com it('cria usuário válido', ...), it('rejeita e-mail duplicado', ...), it('lança ValidationError quando nome está vazio', ...). Maus exemplos: it('test 1'), it('funciona'), it('caso de erro').

Um teste, um comportamento

É tentador empilhar várias verificações dentro de um único it. Resista. Quando o teste falha, você quer saber imediatamente qual comportamento quebrou — não ler 30 linhas para descobrir que o terceiro expect reclamou. Se um cenário de fato precisa verificar três coisas relacionadas, três its separados deixam o relatório mais legível e o erro mais cirúrgico.

Use beforeEach para arranjos repetitivos

Se cinco testes começam exatamente com a mesma preparação ("criar um usuário fake na memória", "limpar a tabela", "abrir transação"), mova para um beforeEach. Mantém os its focados na variação real entre eles.

4. Test doubles: stub, spy, mock, fake

"Mock" virou guarda-chuva no dia a dia, mas a literatura (Gerard Meszaros, Martin Fowler) distingue quatro tipos de test doubles — substitutos que você usa no lugar de um colaborador real para isolar a unidade testada.

TipoPara que serveExemplo
StubRetorna respostas fixas; ignora interaçõesRepository falso que sempre retorna o mesmo usuário
SpyComo o real, mas registra chamadas para inspeçãojest.spyOn(logger, 'info')
MockProgramado com expectativas: "deve ser chamado 1×"expect(repo.save).toHaveBeenCalledWith(user)
FakeImplementação simplificada porém funcionalRepository em memória que simula o banco

Em Jest, todos esses tipos são construídos a partir de jest.fn() e jest.spyOn(). A diferença é a intenção. Stub: você só quer um colaborador que devolva algo previsível. Spy: quer ver se o colaborador foi chamado. Mock: quer afirmar que ele foi chamado de tal jeito. Fake: precisa de comportamento real, mas leve, sem o custo de uma dependência externa.

TypeScript — os quatro doubles em Jest
// stub: resposta fixa
const repoStub = { findById: jest.fn().mockResolvedValue({ id: 1, name: 'Ada' }) };

// spy: observa chamadas reais
const spy = jest.spyOn(console, 'log');

// mock: programado com expectativa
expect(repoStub.findById).toHaveBeenCalledWith(1);

// fake: implementação alternativa
class UserRepoFake {
  private data: Map<number, User> = new Map();
  async save(u: User) { this.data.set(u.id, u); return u; }
  async findById(id: number) { return this.data.get(id) ?? null; }
}
London vs Detroit

Existem duas escolas filosóficas sobre o uso de doubles. A "London school" (a la Steve Freeman) prefere mock-everything: cada colaborador é mockado, o teste afirma interações. A "Detroit school" (a la Kent Beck) prefere usar implementações reais sempre que viáveis e mockar só nas fronteiras (banco, rede). Na prática, a maioria dos times mistura: mocks para colaboradores caros, instâncias reais para classes triviais. Não há resposta certa — o que importa é a coerência dentro do projeto.

5. Configuração — Jest + ts-jest + supertest

O ecossistema TypeScript tem várias opções (Vitest, Mocha, node:test), mas Jest continua sendo a escolha mais difundida por trazer runner, asserções, mocks e cobertura no mesmo pacote. Para usá-lo com TypeScript, basta adicionar ts-jest como transformador. O supertest é o utilitário canônico para testar APIs Express sem subir um servidor real.

Bash — instalação
# dependências de desenvolvimento
npm install -D jest ts-jest @types/jest supertest @types/supertest

# gerar config inicial (opcional)
npx ts-jest config:init
jest.config.ts — mínimo para o projeto
import type { Config } from 'jest';

const config: Config = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  roots: ['<rootDir>/src', '<rootDir>/tests'],
  testMatch: ['**/*.test.ts', '**/*.spec.ts'],
  collectCoverageFrom: [
    'src/**/*.ts',
    '!src/**/*.d.ts',
    '!src/server.ts',
  ],
  coverageDirectory: 'coverage',
  setupFilesAfterEnv: ['<rootDir>/tests/setup.ts'],
  clearMocks: true,
};

export default config;
package.json — scripts
"scripts": {
  "test":           "jest",
  "test:watch":     "jest --watch",
  "test:coverage":  "jest --coverage",
  "test:ci":        "jest --ci --coverage --reporters=default --reporters=jest-junit"
}

Estrutura de pastas sugerida: testes podem ficar colocalizados (ao lado do arquivo testado: format.ts + format.test.ts) ou em uma pasta tests/ espelhando src/. Ambos funcionam; o importante é escolher uma e ser consistente. Colocalização aproxima teste e implementação — quem refatora vê o teste imediatamente.

6. Testando helpers — unit puro

Helpers são o caso mais fácil e o melhor lugar para começar. São funções puras: dado o mesmo input, sempre retornam o mesmo output, sem efeitos colaterais. Não tocam banco, não falam com a rede. Basta importá-las e afirmar.

src/helpers/format.ts
export function formatCurrency(value: number, locale = 'pt-BR'): string {
  return new Intl.NumberFormat(locale, {
    style: 'currency',
    currency: 'BRL',
  }).format(value);
}

export function isValidEmail(input: string): boolean {
  const re = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  return re.test(input.trim().toLowerCase());
}
src/helpers/format.test.ts
import { formatCurrency, isValidEmail } from './format';

describe('formatCurrency', () => {
  it('formata valor positivo', () => {
    expect(formatCurrency(1234.5)).toMatch(/R\$\s*1\.234,50/);
  });

  it('formata zero', () => {
    expect(formatCurrency(0)).toMatch(/R\$\s*0,00/);
  });

  it('formata valor negativo', () => {
    expect(formatCurrency(-10)).toMatch(/-?R\$\s*10,00/);
  });
});

describe('isValidEmail', () => {
  it.each([
    ['a@b.co', true],
    ['   ada@inteli.edu.br ', true],
    ['sem-arroba', false],
    ['a@b', false],
    ['', false],
  ])('%s -> %s', (input, esperado) => {
    expect(isValidEmail(input as string)).toBe(esperado);
  });
});

O it.each é particularmente útil para helpers: gera um caso de teste para cada linha da tabela. É a forma idiomática de testar muitas combinações input/output sem repetir o corpo cinco vezes. Para casos extremos, há o describe.each que aplica o mesmo padrão a um bloco inteiro.

Cuidado com Intl e \u00a0

O Intl.NumberFormat insere um espaço não-quebrável (\u00a0) entre o símbolo da moeda e o número. Se o teste falhar com a mensagem "expected 'R$ 1.234,50' to be 'R$ 1.234,50'" e os strings parecem idênticos, é exatamente isso. Use toMatch com regex (/R\$\s*1\.234,50/) para evitar a armadilha.

7. Testando services — mockando o repository

Services contêm a lógica de negócio: regras de validação, orquestração de várias chamadas, decisões. Eles dependem de repositories para tocar dados, mas a regra de negócio não deveria depender de Postgres estar rodando para ser testada. A solução é mockar o repository.

src/services/UserService.ts
import type { UserRepository } from '../repositories/UserRepository';
import { isValidEmail } from '../helpers/format';

export class EmailDuplicadoError extends Error {}
export class EmailInvalidoError  extends Error {}

export class UserService {
  constructor(private repo: UserRepository) {}

  async create(input: { name: string; email: string }) {
    if (!isValidEmail(input.email)) throw new EmailInvalidoError();
    const existente = await this.repo.findByEmail(input.email);
    if (existente) throw new EmailDuplicadoError();
    return await this.repo.create(input);
  }
}
src/services/UserService.test.ts
import { UserService, EmailDuplicadoError, EmailInvalidoError } from './UserService';
import type { UserRepository } from '../repositories/UserRepository';

function makeRepo(): jest.Mocked<UserRepository> {
  return {
    findByEmail: jest.fn(),
    create:      jest.fn(),
    findById:    jest.fn(),
  } as unknown as jest.Mocked<UserRepository>;
}

describe('UserService.create', () => {
  it('cria usuário válido quando e-mail é único', async () => {
    const repo = makeRepo();
    repo.findByEmail.mockResolvedValue(null);
    repo.create.mockResolvedValue({ id: 1, name: 'Ada', email: 'ada@x.com' });
    const svc = new UserService(repo);

    const r = await svc.create({ name: 'Ada', email: 'ada@x.com' });

    expect(r.id).toBe(1);
    expect(repo.create).toHaveBeenCalledWith({ name: 'Ada', email: 'ada@x.com' });
  });

  it('rejeita e-mail inválido sem chamar repo.create', async () => {
    const repo = makeRepo();
    const svc = new UserService(repo);

    await expect(svc.create({ name: 'X', email: 'invalido' }))
      .rejects.toBeInstanceOf(EmailInvalidoError);
    expect(repo.create).not.toHaveBeenCalled();
  });

  it('rejeita e-mail duplicado', async () => {
    const repo = makeRepo();
    repo.findByEmail.mockResolvedValue({ id: 9, name: 'X', email: 'a@b.com' });
    const svc = new UserService(repo);

    await expect(svc.create({ name: 'Y', email: 'a@b.com' }))
      .rejects.toBeInstanceOf(EmailDuplicadoError);
  });
});

Note três coisas. Primeiro, o teste nunca precisa de Postgres rodando — todos os efeitos colaterais foram simulados com jest.fn(). Segundo, o teste afirma comportamento ("não chama repo.create quando e-mail é inválido"), não implementação interna. Terceiro, o tipo jest.Mocked<UserRepository> dá auto-complete e segurança: se você renomear um método do repo, o teste estoura no compilador antes de rodar.

Não mocke o que você está testando

Erro clássico do iniciante: testar UserService.create e mockar UserService no caminho. Aí o teste passa sem nem executar a unidade real. Mocke os colaboradores do sujeito (repo, logger, e-mail), nunca o sujeito em si.

8. Testando repositories — banco efêmero

Repositories tocam o banco. Mocká-los aqui esvazia o teste — o objetivo é justamente verificar que o SQL escrito faz o que se espera. A pergunta é: como rodar testes contra Postgres sem precisar de um banco compartilhado, sem deixar rastros entre testes e sem demorar minutos?

Há três caminhos viáveis no nosso stack:

  • Postgres em Docker via docker compose. Sobe um container postgres:16-alpine antes da suíte e derruba ao final. Rápido (~1s para subir), idêntico à produção.
  • Testcontainers (biblioteca testcontainers no npm). Sobe containers programaticamente dentro do próprio teste. Útil em CI/CD onde o Docker já está disponível.
  • pg-mem: implementação em memória (JavaScript) compatível com a maior parte do dialeto Postgres. Não tem todos os recursos do Postgres real (extensões, alguns tipos exóticos), mas é instantâneo.

O segredo do isolamento entre testes é abrir uma transação no beforeEach e dar ROLLBACK no afterEach. Cada teste enxerga um banco "limpo" sem precisar truncar tabelas — e nenhum dado vaza para o próximo.

tests/setup.ts — pool global
import { Pool } from 'pg';

export const pool = new Pool({
  host: process.env.PGHOST ?? 'localhost',
  port: Number(process.env.PGPORT ?? 5432),
  database: process.env.PGDATABASE ?? 'app_test',
  user: process.env.PGUSER ?? 'postgres',
  password: process.env.PGPASSWORD ?? 'postgres',
});

afterAll(async () => { await pool.end(); });
src/repositories/UserRepository.test.ts
import { pool } from '../../tests/setup';
import { UserRepository } from './UserRepository';
import type { PoolClient } from 'pg';

describe('UserRepository (integração)', () => {
  let client: PoolClient;
  let repo: UserRepository;

  beforeEach(async () => {
    client = await pool.connect();
    await client.query('BEGIN');
    repo = new UserRepository(client);
  });

  afterEach(async () => {
    await client.query('ROLLBACK');
    client.release();
  });

  it('create + findByEmail retornam o mesmo registro', async () => {
    const u = await repo.create({ name: 'Ada', email: 'ada@x.com' });
    const achado = await repo.findByEmail('ada@x.com');
    expect(achado?.id).toBe(u.id);
  });

  it('findByEmail retorna null quando não existe', async () => {
    expect(await repo.findByEmail('fantasma@x.com')).toBeNull();
  });

  it('create estoura ao violar UNIQUE de email', async () => {
    await repo.create({ name: 'A', email: 'dup@x.com' });
    await expect(repo.create({ name: 'B', email: 'dup@x.com' }))
      .rejects.toThrow(/duplicate key/i);
  });
});

Como a transação é revertida no afterEach, o teste create estoura ao violar UNIQUE não polui o próximo. Você pode rodar a suíte em paralelo (jest --maxWorkers=4) que cada worker terá sua própria conexão e seu próprio rollback. Bancos de teste não precisam ser frágeis.

Migrations no setup global

Para testes de integração funcionarem, o schema precisa existir. A prática consagrada é rodar as migrations reais da aplicação contra o banco de teste no globalSetup do Jest. Assim o esquema testado é exatamente o que vai pra produção, e migrar a aplicação implica revalidar os testes.

9. Testando controllers — supertest

Controllers fazem o "casamento" entre HTTP e domínio: pegam req.body, chamam o service, traduzem o resultado em res.status(...).json(...) ou res.render(...). Testá-los unitariamente (mockando req/res) é tedioso e quebradiço. O caminho idiomático é integração leve: instanciar o app Express e usar supertest para disparar requisições HTTP em memória.

src/app.ts — exportar o Express
import express from 'express';
import { userRouter } from './controllers/UserController';

export function buildApp() {
  const app = express();
  app.use(express.json());
  app.set('view engine', 'ejs');
  app.use('/api/users', userRouter);
  return app;
}
tests/users.controller.test.ts
import request from 'supertest';
import { buildApp } from '../src/app';

const app = buildApp();

describe('POST /api/users', () => {
  it('cria usuário e retorna 201', async () => {
    const res = await request(app)
      .post('/api/users')
      .send({ name: 'Ada', email: 'ada@x.com' })
      .expect(201);

    expect(res.body).toEqual(expect.objectContaining({
      id: expect.any(Number),
      email: 'ada@x.com',
    }));
  });

  it('retorna 400 com email inválido', async () => {
    await request(app)
      .post('/api/users')
      .send({ name: 'X', email: 'invalido' })
      .expect(400);
  });

  it('retorna 409 quando email duplicado', async () => {
    await request(app)
      .post('/api/users')
      .send({ name: 'A', email: 'dup@x.com' });

    await request(app)
      .post('/api/users')
      .send({ name: 'B', email: 'dup@x.com' })
      .expect(409);
  });
});

Importante: supertest(app) não precisa de servidor escutando em porta. Ele despacha as requisições direto contra a instância do Express, em memória. Isso faz com que cada teste leve ~10-30ms — rápido o bastante para rodar dezenas em segundos.

Injetar dependências para testar sem Postgres

Para o teste de controller acima rodar sem banco, você pode usar uma fake do repository (em memória). Para isso, exporte o buildApp aceitando dependências, ou aplique injeção via container de DI. Em projetos pequenos, basta uma fábrica:

src/app.ts — versão com DI explícita
export function buildApp(deps: { userService: UserService }) {
  const app = express();
  app.use(express.json());
  app.use('/api/users', makeUserRouter(deps.userService));
  return app;
}

10. Smoke test em views EJS

Views EJS renderizadas no servidor (SSR) são strings de HTML. Testá-las unitariamente é raramente útil — o template é simples; o que importa é se ele recebeu os dados certos. A estratégia mais barata é o smoke test: bater no controller que renderiza a página, capturar a resposta HTML e verificar via regex se aparece algo crítico (o <title>, um nome, uma classe).

tests/users.view.test.ts
import request from 'supertest';
import { buildApp } from '../src/app';

const app = buildApp(/* deps fake */);

describe('GET /users (view)', () => {
  it('renderiza HTML com título e tabela', async () => {
    const res = await request(app).get('/users').expect(200);
    expect(res.text).toMatch(/<title>Usuários/);
    expect(res.text).toMatch(/<table[^>]*class="[^"]*users-table/);
    expect(res.headers['content-type']).toMatch(/text\/html/);
  });
});

Esse tipo de teste pega a maior parte das regressões de view: template renomeado, variável faltando, layout que sumiu. O que ele não pega é interação no browser real — para isso, existe E2E.

E2E real com Playwright

Quando o fluxo importa demais para confiar só em regex (cadastro, checkout, login), entra Playwright. Ele controla um navegador real, preenche campos, clica em botões e verifica o resultado de verdade. É o mesmo Playwright que o repositório de aulas usa em tests/index.spec.ts.

tests/e2e/cadastro.spec.ts
import { test, expect } from '@playwright/test';

test('usuário consegue se cadastrar', async ({ page }) => {
  await page.goto('http://localhost:3000/cadastro');
  await page.getByLabel('Nome').fill('Ada Lovelace');
  await page.getByLabel('E-mail').fill('ada@x.com');
  await page.getByRole('button', { name: 'Cadastrar' }).click();

  await expect(page.getByText('Cadastro realizado')).toBeVisible();
});
Não exagere no E2E

Cada teste E2E custa segundos. Se sua suíte E2E passa de 5 minutos, ela vai ser pulada pelo time. Mantenha E2E só para os fluxos críticos do produto. Se você está testando casos de borda no E2E ("e se o telefone tiver 14 dígitos?"), provavelmente esse caso pertence a um teste unitário do helper de validação.

11. Models — o compilador já testa

No nosso stack, "models" são tipos TypeScript: interface User, type CreateUserInput, enum Status. Eles não têm comportamento — descrevem dados. Testar tipos com Jest seria redundante: o compilador já faz esse trabalho a cada tsc --noEmit.

src/models/User.ts
export interface User {
  id:        number;
  name:      string;
  email:     string;
  createdAt: Date;
}

export type CreateUserInput = Omit<User, 'id' | 'createdAt'>;

Se um service tentar criar um User sem email, o build estoura. Se um repository devolver string onde se esperava Date, idem. Por isso é tão importante que o pipeline de CI rode tsc --noEmit junto dos testes — é parte da bateria de qualidade, ainda que ninguém escreva um describe sobre tipos.

Quando testar model com runtime check

Se você usa uma biblioteca como Zod ou io-ts para validação em runtime (ex.: validar JSON da requisição), aí sim faz sentido escrever testes: garanta que o schema rejeita o que deveria rejeitar e aceita o que deveria aceitar.

12. Cobertura — boa métrica, péssima meta

Cobertura de testes (coverage) mede quais linhas, branches ou funções foram executadas durante a suíte. Em Jest, --coverage gera um relatório por arquivo com quatro métricas:

MétricaO que conta
Statements% de instruções executadas
Branches% de ramos (if/else, &&, ||) atingidos
Functions% de funções chamadas
Lines% de linhas executadas
Bash — rodar com cobertura
npm test -- --coverage

# abre o relatório HTML detalhado
open coverage/lcov-report/index.html

Cobertura é uma métrica útil para encontrar buracos: se um arquivo crítico está com 12% de branches, há cenários inteiros sem teste. Mas é uma péssima meta. Cravar "100% de cobertura" empurra o time a escrever testes triviais que só executam o código sem afirmar comportamento — atinge a métrica e não pega bug nenhum.

A armadilha do 100%

Já vi times exigirem 100% de cobertura via política de CI e a primeira coisa que aparece é o "teste decorativo": chamar a função sem expect, marcar a linha como coberta e seguir adiante. A métrica subiu, a qualidade caiu. Trate cobertura como termômetro, nunca como vacina.

Uma régua razoável: 80% de statements nas pastas de services e helpers, 60% nos controllers (porque parte deles é "casing" trivial), e 0% obrigado em scripts auxiliares (excluídos via collectCoverageFrom).

13. TDD — red, green, refactor

TDD (Test-Driven Development) é o ciclo curto proposto por Kent Beck: escreva o teste antes da implementação, veja-o falhar (vermelho), faça-o passar com o mínimo de código (verde) e só então melhore o código sem mudar comportamento (refactor). É uma disciplina, não uma religião — vale aplicar onde o domínio compensa.

  1. Red: escreva um teste que descreva uma capacidade ainda inexistente. Ele falha porque a função sequer existe.
  2. Green: implemente o mínimo para o teste passar. Pode ser feio. Pode ter if hardcoded. Foco é virar o teste para verde.
  3. Refactor: com o teste verde como rede de segurança, melhore o código — extraia funções, renomeie variáveis, simplifique condicionais. O teste continua verde porque o comportamento não mudou.
TDD — passo a passo de um helper
// 1. RED — teste falha porque slugify não existe
describe('slugify', () => {
  it('troca espaços por hífens e baixa caixa', () => {
    expect(slugify('Olá Mundo')).toBe('ola-mundo');
  });
});

// 2. GREEN — implementação mínima para virar verde
export function slugify(s: string) {
  return s.toLowerCase().normalize('NFD')
          .replace(/[\u0300-\u036f]/g, '')
          .replace(/\s+/g, '-');
}

// 3. REFACTOR — melhora sem mudar comportamento
const RE_DIACRITICOS = /[\u0300-\u036f]/g;
const RE_ESPACOS     = /\s+/g;
export function slugify(s: string) {
  return s.toLowerCase()
          .normalize('NFD')
          .replace(RE_DIACRITICOS, '')
          .replace(RE_ESPACOS, '-');
}

Quando funciona bem? Em lógica de negócio com regras claras (validações, cálculos, regras de domínio). Quando não compensa? Em código exploratório onde a interface ainda muda a cada hora — você gastará mais tempo reescrevendo testes do que implementação.

14. Husky e lint-staged — gate local antes do commit

Antes do commit chegar ao CI remoto, há uma camada barata de validação: o pre-commit hook. Ele roda na máquina do desenvolvedor, em segundos, e impede que código quebrado seja commitado. husky instala hooks via npm; lint-staged roda comandos só nos arquivos modificados (não na base inteira).

Bash — instalação
npm install -D husky lint-staged
npx husky init
echo "npx lint-staged" > .husky/pre-commit
package.json — config lint-staged
"lint-staged": {
  "*.ts": [
    "eslint --fix",
    "tsc-files --noEmit"
  ],
  "*.{json,md}": ["prettier --write"]
}

Resultado: a cada git commit, os arquivos staged são corrigidos pelo ESLint, validados pelo compilador TS e formatados. Se algum passo falhar, o commit é abortado. O autor descobre o problema em segundos, não em 5 minutos esperando o CI.

Não rode a suíte inteira no pre-commit

É tentador colocar npm test no pre-commit. Não faça. Para projetos com mais de 50 testes, isso vira tempo de espera no commit e gente passa a usar --no-verify. Lint + types já filtram a maior parte dos erros bobos; deixe a suíte completa para o CI ou para um pre-push opcional.

15. GitHub Actions — CI no remoto

O CI (Continuous Integration) é o robô que pega seu push, instala dependências, roda lint, testes, build, e devolve um sinal verde ou vermelho no PR. Em GitHub, isso é configurado via arquivos YAML em .github/workflows/. Cada arquivo define um workflow; cada workflow tem jobs; cada job, steps.

.github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB:       app_test
        ports: ['5432:5432']
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Type-check
        run: npx tsc --noEmit

      - name: Test (com Postgres como service)
        env:
          PGHOST: localhost
          PGUSER: postgres
          PGPASSWORD: postgres
          PGDATABASE: app_test
        run: npm test -- --ci --coverage

O bloco services.postgres é a magia: o GitHub Actions sobe um container Postgres ao lado do job, expõe a porta 5432, e a aplicação se conecta como se fosse local. Custa zero configuração extra. Para Redis, MongoDB, etc., o padrão é o mesmo.

Status checks no PR

Configure no GitHub: Settings → Branches → Branch protection rules → Require status checks to pass. Marque o job test. A partir daí, ninguém faz merge para main com CI vermelho — nem o owner do repo. É a versão do projeto da regra "vermelho não passa".

Caching agressivo

Adicionar cache: 'npm' ao setup-node reduz o tempo de instalação de ~50s para ~5s em PRs subsequentes. Para builds de TypeScript pesados, o action actions/cache pode armazenar dist/ entre runs. CI rápido = time confiando no CI.

16. Boas práticas — testes que envelhecem bem

  • Determinísticos: mesmo input, mesmo resultado, sempre. Se um teste depende de Date.now(), mocke. Se depende de Math.random(), injete uma seed.
  • Independentes: a ordem de execução não pode importar. Cada teste arranja seu próprio cenário e limpa quando termina.
  • Rápidos: unitários em milissegundos, integração em centenas de ms. Se um teste leva 10s, virá moroso o suficiente para ninguém esperar.
  • Auto-explicativos: um humano lê o nome e entende o cenário. Sem precisar abrir o corpo.
  • Focados em comportamento, não implementação: o teste sobrevive a refactor. Se você renomeou um método privado e o teste quebrou, ele estava verificando a implementação errada.
  • Mockando colaboradores, não o sujeito: mocke as dependências externas, deixe o objeto sob teste real.
  • Um expect primário por it: múltiplas verificações relacionadas tudo bem; mas se vai testar três comportamentos, três its.
  • Falham por uma boa razão: se um teste falha, ele aponta o defeito. Se ele falha por motivos cosméticos (espaços em branco, ordem de chaves), recalibre o asserter.

17. Anti-padrões — testes que viram lastro

Flakey tests

Testes que passam às vezes e falham às vezes — sem mudança no código. Causas comuns: dependência de horário (new Date()), de rede (chamada a um endpoint externo), de ordem (teste B precisa que A tenha rodado antes), de timing assíncrono (setTimeout sem await apropriado). O custo de um flakey é altíssimo: cada PR vermelho falsamente desbasta a confiança no CI. Se um teste é flakey duas vezes, conserte ou apague — não deixe de quarentena.

Mocks demais

Sintoma: o teste se torna um espelho da implementação. Você muda a ordem de duas chamadas internas e o teste quebra, mesmo que o comportamento externo seja idêntico. O remédio é mockar só o que sai do limite do projeto (banco, rede, sistema de arquivos) e usar instâncias reais para tudo dentro.

Dependência de ordem

Quando o teste C só passa porque B inseriu o registro X. Em Jest, basta um --randomize ou paralelismo para descobrir. Cura: cada teste cria/derruba seu próprio cenário (transação + rollback, fixtures por beforeEach, banco em memória recriado).

Testes "decorativos"

Aqueles que executam o código sem verificar nada relevante: expect(typeof x).toBe('object'). Sobem cobertura, não pegam bug. Veja se cada expect realmente afirma um comportamento; se for só "não explodiu", remova.

Testar implementação privada

Espionar métodos internos da classe, afirmar a ordem de chamadas particulares — tudo isso amarra o teste à implementação. Refactor inocente quebra cinquenta testes. Teste pela API pública.

A regra do "se eu deletar isso, o teste continua passando?"

Antes de comitar um teste novo, faça este experimento mental: se eu apagar uma linha do código de produção, esse teste falha? Se a resposta for não, o teste é decorativo. Apague-o ou reescreva-o até que apagar produção produza vermelho.
