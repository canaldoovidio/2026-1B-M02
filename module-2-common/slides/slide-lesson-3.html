⚡ Quiz socrático — 2 de 3
A modelagem como ferramenta crítica — não decorativa
Pergunta 2 · ORM · Bugayenko

"Em qual momento da modelagem do BrPec um ORM nos faria perder informação?"

📚 Autoestudo: ORM is an Offensive Anti-Pattern

🎯 Âncora da resposta

ORM esconde o SQL. Funciona pra CRUD trivial; quebra em junções multinível, agregações e constraints declarativas.

Aplicado ao BrPec: exportar movimentações por retiro/período no relatório do Coordenador vira N+1 queries com ORM ingênuo — destrói a janela de upload do offline-first. Em SQL puro: 1 SELECT com GROUP BY.

Nesta aula escrevemos SQL à mão para enxergar o que o ORM esconde.

⚡ Quiz socrático — 3 de 3
A diferença que define se a modelagem é mecânica ou imprecisa
Pergunta 3 · Chen · 1976

"Qual a diferença entre um diagrama ER e um DER, em uma frase?"

📚 Autoestudo: Entity-Relationship Modeling: Historical Events, Future Trends, and Lessons Learned

🎯 Âncora da resposta

ER é o esboço da história — quais entidades existem e como se conectam. DER é o storyboard rotulado — atributos explícitos, PK/FK e cardinalidades exatas.

Aplicado ao BrPec: ER diz "Capataz registra Evento em Retiro". DER diz "Capataz(id, nome) ←FK— Evento(id, data, capataz_id, retiro_id) —FK→ Retiro(id)".

Por que importa: a passagem ER→DER é onde a rastreabilidade RN→Tabela é cravada. Sem DER, integridade vira "vou validar no JS".

🗝️ Diagrama DER
Cardinalidades explícitas · PK em sublinhado · FK identificada
Capatazid, nome
Eventoid, tipo, data, capataz_id↗, retiro_id↗, bovino_id↗
Retiroid, nome

↗ = chave estrangeira. 1 capataz registra N eventos. 1 retiro recebe N eventos. 1 bovino tem N eventos.

💾 Modelo físico — DDL em SQLite
Tradução mecânica do DER · cada cardinalidade vira uma FOREIGN KEY
CREATE TABLE retiros (
  id        INTEGER PRIMARY KEY,
  nome      TEXT    NOT NULL UNIQUE,
  municipio TEXT    NOT NULL
);

CREATE TABLE capatazes (
  id        INTEGER PRIMARY KEY,
  nome      TEXT    NOT NULL,
  retiro_id INTEGER NOT NULL REFERENCES retiros(id)
);

CREATE TABLE bovinos (
  id        INTEGER PRIMARY KEY,
  brinco    TEXT    NOT NULL UNIQUE,
  sexo      TEXT    NOT NULL CHECK(sexo IN ('M','F')),
  retiro_id INTEGER NOT NULL REFERENCES retiros(id)
);

CREATE TABLE eventos_zootecnicos (
  id         INTEGER PRIMARY KEY,
  tipo       TEXT    NOT NULL CHECK(tipo IN ('NASCIMENTO','MORTE','VENDA','COMPRA','TRANSFERENCIA')),
  data       TEXT    NOT NULL,        -- ISO-8601
  bovino_id  INTEGER NOT NULL REFERENCES bovinos(id),
  retiro_id  INTEGER NOT NULL REFERENCES retiros(id),
  capataz_id INTEGER NOT NULL REFERENCES capatazes(id),
  observacao TEXT
);

Cada constraint cumpre uma RN explícita. Sem isso, integridade vira "promessa" da camada de aplicação.

📐 Cardinalidade · 1 : N
A relação dominante na modelagem do BrPec — leia a seta

1 capataz registra N eventos. Mas cada evento tem exatamente 1 capataz responsável.

No DDL: a tabela eventos_zootecnicos tem a coluna capataz_id (lado N). A FK mora sempre no lado N.

Quando aparece, exige tabela de junção

1 tarefa pode ser atribuída a vários capatazes. 1 capataz pode receber várias tarefas. Sem tabela intermediária, é impossível modelar.

📐 Cardinalidade · 1 : 1
Rara — quase sempre cabe numa única tabela

Ficha Sanitária

Pergunte sempre: isso é mesmo 1:1 ou pode virar 1:N no futuro?

Se for genuinamente 1:1 e os atributos são consultados juntos, colapse em 1 só tabela. Separe só se houver razão de ciclo de vida ou segurança.

⚡ CRUD ao vivo — rastreabilidade em ação
Pegamos a tabela de 1 grupo voluntário · cada operação dispara 1 pergunta socrática
INSERT
Create

Se eu omitir capataz_id, o que o banco faz?

SELECT
Read

Como garanto que esse SELECT é rápido com 100k linhas?

UPDATE
Update

O que acontece se eu esquecer o WHERE? — demonstrar.

DELETE
Delete

Por que soft delete (deleted_at) é comum em fazendas?

-- INSERT — registra o nascimento do Cleber via rádio
INSERT INTO eventos_zootecnicos (tipo, data, bovino_id, retiro_id, capataz_id, observacao)
VALUES ('NASCIMENTO', '2026-05-05', 47, 3, 12, 'parto assistido');

-- SELECT — relatório do Coordenador para o retiro Aroeira em maio
SELECT tipo, data, bovino_id FROM eventos_zootecnicos
WHERE retiro_id = 3 AND data >= '2026-05-01'
ORDER BY data DESC;

-- UPDATE — corrigir observação. ATENÇÃO: SEM o WHERE altera tudo.
UPDATE eventos_zootecnicos SET observacao = 'parto natural, mãe saudável'
WHERE id = 1;

-- DELETE — soft delete preserva auditoria zootécnica
UPDATE eventos_zootecnicos SET deleted_at = '2026-05-05T10:00' WHERE id = 1;

Princípio do dia: toda mutação precisa ser precedida pelo SELECT que prova o que vai mudar.
