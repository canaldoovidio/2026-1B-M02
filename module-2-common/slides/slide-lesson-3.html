📋 Contexto da atividade
De atores e regras de negócio para design computacional verificável
Aula 1 + Aula 2 → entrada RNFs técnicos verificáveis Diagramas de sequência UML

Nas aulas 1 e 2 você mapeou atores, regras de negócio e entidades do seu projeto.

Agora é hora de traduzir isso em design computacional: descrever os RNFs de forma técnica e verificável, e representar os fluxos principais como diagramas de sequência UML entre as camadas do back-end.

🎯 O que esta atividade entrega para o seu projeto

  • RNFs deixam de ser frases vagas e viram métricas, critérios e checklists que cabem em testes (ponte direta para a aula 10).
  • Os fluxos do sistema ganham contrato visual entre Controller, Service, Repository e Banco — a mesma arquitetura MVC de 6 camadas das aulas 5–6.
  • Cada decisão técnica fica rastreada até o RF que a justifica — base para argumentar trade-offs em revisão.

Esta é a viewpoint Computacional do RM-ODP aplicada ao seu projeto: como o sistema é decomposto em componentes que conversam.

📦 Entrega mínima
O que precisa estar presente para a atividade ser considerada — não garante nota máxima
  • Tabela de RNFs preenchida para os 8 Eixos, com ao menos uma métrica ou critério concreto por eixo.
  • 3 diagramas de sequência com as 4 camadas (Controller → Service → Repository → Banco), fluxo principal e ao menos 1 fluxo alternativo cada.
  • Tabela de rastreabilidade (RF → RNF → Diagrama) preenchida para os 3 diagramas.
Critério de qualidade: se o RNF não puder ser testado ou inspecionado, reescreva até que possa.

✅ Boa métrica

"Resposta de GET /pedidos < 300 ms no p95 com 100 pedidos no banco."

✅ Bom critério

"Toda query do repository usa parâmetros $1 — verificável por revisão de código."

❌ Vago demais

"O sistema deve ser rápido e seguro."
