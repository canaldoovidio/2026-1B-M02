Por que um dev de aplicação precisa entender redes?
Você programa HTTP — mas é TCP/IP que entrega o pacote

🧠 O que você não vê

Quando você chama res.render('home'), o navegador do usuário acabou de passar por:

  • Resolução DNS de app.empresa.com.br
  • 3-way handshake TCP
  • Handshake TLS (HTTPS)
  • Possível CDN no caminho
  • Load balancer roteando para o seu servidor

🎯 O que muda no seu código

  • Cache HTTP bem configurado economiza 90% das requisições.
  • Timeout em chamada externa evita travar o Express inteiro.
  • Retry com backoff sobrevive a falhas momentâneas.
  • Idempotência impede cobrança duplicada.
  • Logs e métricas permitem encontrar o gargalo.

💡 Tese da aula

Redes não são "problema do DevOps". O dev backend que entende TCP, DNS, TLS, cache e tolerância a falhas escreve software significativamente mais robusto e barato de operar.

Modelo TCP/IP — 4 Camadas
Cada camada esconde a complexidade da camada de baixo

🎯 Para o dev de aplicação

Você toca a camada 4 (Application) diretamente, depende implicitamente da 3 (Transport) para confiabilidade, e quase nunca pensa nas camadas 1 e 2 — até elas falharem.

TCP × UDP · 3-way Handshake
Confiabilidade vs. velocidade — escolha consciente
TCP
Confiável, ordenado, com handshake
3-way handshake (SYN → SYN-ACK → ACK), retransmissão, controle de congestionamento. Base do HTTP/1.1 e HTTP/2.

UDP

Sem handshake, sem retransmissão. Perde pacotes silenciosamente — útil quando latência > perfeição.

DNS — A Lista Telefônica da Internet
Tradução de nome (app.empresa.com.br) → IP (203.0.113.42)

🏷️ Tipos de Registro

  • A · nome → IPv4
  • AAAA · nome → IPv6
  • CNAME · nome → outro nome (alias)
  • MX · servidor de e-mail do domínio
  • TXT · texto livre (SPF, DKIM, verificações)
  • NS · servidores de nome autoritativos
  • SOA · zona DNS — serial, TTL, autoridade

🔍 Resolução de um nome

  1. Cache do navegador
  2. Cache do sistema operacional
  3. Resolver recursivo (ISP / 1.1.1.1 / 8.8.8.8)
  4. Servidor raiz (.)
  5. Servidor TLD (.br)
  6. Servidor autoritativo (empresa.com.br)
  7. Resposta + cache pelo TTL

⏱️ TTL e Propagação

TTL define por quantos segundos o registro pode ser cacheado. Antes de mudar DNS de produção, baixe o TTL para 60s com 24h de antecedência — assim a troca propaga rápido.

🛠️ Ferramentas

dig app.empresa.com.br · nslookup app.empresa.com.br · dig +trace mostra a resolução completa em cascata.

HTTP — três gerações em produção hoje
1.1, 2 e 3 convivem; o cliente negocia a melhor versão suportada
HTTP/1.1
Texto, uma req por vez na conexão
Headers em texto plano, head-of-line blocking. Persistente com Keep-Alive. Ainda é a base mental para entender o resto.
1997 · TCP · TLS opcional
HTTP/2
Binário, multiplexação, header compression
Várias requests simultâneas em uma única conexão TCP. HPACK comprime headers. Server Push (raramente usado).
2015 · TCP + TLS · ALPN h2
HTTP/3
Sobre QUIC (UDP) — sem TCP
Elimina head-of-line blocking de transporte. Handshake e TLS 1.3 fundidos. Conexão sobrevive a troca de IP (mobile).
2022 · UDP · QUIC · ALPN h3

⚠️ Por que importa

Sem HTTPS: senhas, cookies de sessão, tokens trafegam em texto puro. Qualquer roteador no caminho lê e modifica. Em 2026, HTTPS é não-negociável.

🪜 Camadas de cache no caminho

Navegador → CDN → Reverse proxy (nginx) → Aplicação (cache em memória) → Banco. Cada camada que responde é uma camada que o seu Express não precisou tocar.

CDN — Servindo do mais perto possível
Origem fica em um lugar; o conteúdo, em centenas de edges

🏛️ Origem

Seu servidor Express em São Paulo. Fonte da verdade.

🌍 Edge / PoP

Centenas de pontos no mundo. Cacheiam estáticos e HTML elegível.

📱 Usuário

Em Lisboa, Tóquio, Nova York. Recebe do edge mais próximo.

✅ Cache HIT

Edge tem o recurso fresco. Responde direto, em ms. Origem não é tocada.

❌ Cache MISS

Edge busca na origem, guarda e responde. Próxima vez vira HIT.

🧹 Purge

Invalida cache no edge após deploy. API do CDN ou versão no nome do arquivo (app.a1b2.js).

🛒 No mercado

Cloudflare · Fastly · AWS CloudFront · Akamai. Hoje uma CDN também roda edge functions (lógica perto do usuário) e oferece WAF, DDoS protection e análise de tráfego.

Tolerância a Falhas · Retry, Timeout, Idempotência
A rede sempre falha. A pergunta é: como o seu código reage?
timeout-retry.ts
// timeout no fetch interno entre serviços
async function callPayment(orderId: string) {
const ctrl = new AbortController();
const t = setTimeout(() => ctrl.abort(), 3000);
try {
const res = await fetch(url, {
method: 'POST',
headers: { 'Idempotency-Key': orderId },
body: JSON.stringify({ orderId }),
signal: ctrl.signal,
});
return await res.json();
} finally { clearTimeout(t); }
}
TIMEOUT
Sempre. Sem exceção.
Sem timeout, uma chamada lenta trava a thread do Node por minutos. Morte por mil cortes.
RETRY + BACKOFF
Tente de novo, mas espere mais
200ms, 400ms, 800ms · com jitter (ruído aleatório) para evitar thundering herd.
IDEMPOTENCY-KEY
Mesma chave = mesma operação
Servidor remoto sabe que é uma retentativa, não uma compra duplicada.
FALLBACK
Plano B em caso de falha
Retorne dados em cache, mensagem amigável ou degradação parcial — nunca uma stack trace.
Circuit Breaker + Observabilidade RED
Pare de bater na porta fechada · meça o que importa

🛡️ Circuit Breaker — conceito

Como o disjuntor da sua casa: depois de N falhas seguidas, o circuito abre e para de tentar — protegendo o sistema do qual depende.

  • Closed · tudo normal, requests passam
  • Open · falhou demais, retorna erro imediato (sem chamar)
  • Half-Open · após cooldown, tenta uma — se ok, fecha; se falha, abre de novo

🚫 Anti-padrões

  • Retry infinito (sem teto)
  • POST sem Idempotency-Key sendo retried
  • Chamada externa sem timeout
  • Logar segredo no caderno de bordo
