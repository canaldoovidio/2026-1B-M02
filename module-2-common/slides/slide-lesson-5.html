PWA · três pilares

Uma página web que ganha 3 superpoderes — e vira "quase nativo".

🔒

HTTPS (ou localhost)

Sem HTTPS, navegador bloqueia SW e crypto.subtle. Em produção é obrigatório — GitHub Pages serve via HTTPS, ok.

  • Em dev: localhost é exceção válida.
📄

Manifest (.webmanifest)

JSON que descreve o app: nome, ícones, cor de tema, modo de exibição. É o que aparece quando o navegador oferece "Instalar".

  • display: standalone esconde a barra do browser.
⚙️

Service Worker

Script que vive fora da página. Intercepta requisições, cacheia, responde offline. É o coração do offline-first.

  • Roda em thread própria — não bloqueia a UI.

Lighthouse PWA score (que vamos avaliar na ponderada) é basicamente: você tem os 3 pilares funcionando + alguns detalhes (ícone 192x192, start_url cacheável, etc).

Manifest · o cartão de visita do app

Arquivo JSON declarativo. Sem regra de negócio, só metadados.

// manifest.webmanifest
{
  "name":        "BrPec — Coletor de Eventos Zootécnicos",
  "short_name":  "BrPec",
  "start_url":   "./index.html",
  "display":     "standalone",
  "orientation": "portrait",
  "background_color": "#0f172a",
  "theme_color":      "#0d9488",
  "lang": "pt-BR",
  "icons": [
    {
      "src": "./icon.svg",
      "sizes": "any",
      "type": "image/svg+xml",
      "purpose": "any maskable"
    }
  ]
}

O que cada campo controla

  • name — vira o título no app drawer e splash.
  • short_name — fallback quando o nome é cortado.
  • start_url — página de entrada quando o app abre.
  • display:standalone — esconde URL/abas → parece nativo.
  • theme_color — cor da barra de status no Android.
  • icons — Android usa maskable; iOS usa apple-touch-icon.

Conectado no HTML: <link rel="manifest" href="./manifest.webmanifest">.

// sw.js — esqueleto
const CACHE = 'brpec-pwa-v1';
const SHELL = ['./', './index.html', './app.js', './styles.css', './manifest.webmanifest'];

self.addEventListener('install',  (e) => e.waitUntil(caches.open(CACHE).then((c) => c.addAll(SHELL))));
self.addEventListener('activate', (e) => e.waitUntil(self.clients.claim()));
self.addEventListener('fetch',    (e) => e.respondWith(caches.match(e.request).then((r) => r || fetch(e.request))));

Estratégias de cache · qual usar onde?

A regra: shell estática usa cache-first; dados frescos usam network-first.

cache-first · para a shell

Tenta o cache. Se achou, devolve. Senão, busca rede e cacheia para a próxima.

caches.match(e.request).then((c) =>
  c || fetch(e.request).then((r) => {
    caches.open(CACHE).then((b) => b.put(e.request, r.clone()));
    return r;
  })
);

Bom para: HTML, CSS, JS, wasm, ícones, brpec.db (seed).

network-first · para dados

Tenta a rede primeiro. Se falhou, devolve o cache (ou erro estruturado).

fetch(e.request).catch(() =>
  new Response(
    JSON.stringify({ ok: false, reason: 'offline' }),
    { status: 503, headers: { 'content-type': 'application/json' } }
  )
);

Bom para: /sync/* — você quer fresh, mas degradação graciosa.

O SW da PWA do BrPec faz exatamente isso: SHELL_ASSETS em cache-first; /sync/* em network-first com fallback 503 JSON estruturado.

sql.js + IndexedDB · banco persistente no navegador

Evolução do sandbox da aula 4: o blob do banco vive no IndexedDB. Não some no reload.

Fluxo de boot (3 passos)

  1. Motor — initSqlJs() carrega o WASM.
  2. Blob — tenta ler do IDB (key brpec.db); se vazio, fetch o seed inicial e grava no IDB.
  3. Banco — new SQL.Database(blob) abre em memória.

Após cada mutação: db.export() serializa o estado → grava no IDB. Próximo boot reabre exatamente onde parou.

// db.js — núcleo da persistência
async function initDb() {
  const SQL = await initSqlJs({
    locateFile: (f) => `cdn/${f}`
  });

  let blob = await idbGet('brpec.db');
  if (!blob) {
    blob = await fetchSeed();
    await idbSet('brpec.db', blob);
  }

  const db = new SQL.Database(blob);

  async function persist() {
    await idbSet('brpec.db', db.export());
  }

  return { db, persist };
}

Por que IndexedDB e não OPFS? OPFS exige crossOriginIsolated (COOP/COEP), que o GitHub Pages não serve. IndexedDB funciona em qualquer host com qualquer header. Portabilidade ganha.

UUID v4 · gerado no celular do capataz

Por que não deixar o servidor gerar o id? Porque o servidor pode estar a 4 horas de distância.

O contrato-chave

Cada evento, no momento da coleta, recebe um client_uuid gerado localmente. Ele acompanha o evento toda vez que ele atravessa a rede.

  • Sem internet, o registro já tem id estável.
  • Reenvios usam o mesmo id — servidor reconhece e ignora.
  • Auditoria liga o registro no celular ao registro no SGBD.

Geração: crypto.randomUUID() em contextos seguros; fallback crypto.getRandomValues em qualquer outro.

// portable UUID v4
function makeUUID() {
  if (crypto.randomUUID) return crypto.randomUUID();

  const b = new Uint8Array(16);
  crypto.getRandomValues(b);
  b[6] = (b[6] & 0x0f) | 0x40;  // version 4
  b[8] = (b[8] & 0x3f) | 0x80;  // variant

  const h = [...b].map((x) => x.toString(16).padStart(2, '0'));
  return `${h[0]}${h[1]}${h[2]}${h[3]}-${h[4]}${h[5]}-${h[6]}${h[7]}-${h[8]}${h[9]}-${h.slice(10).join('')}`;
}

Idempotência · "fazer X duas vezes vira fazer X uma vez"

A propriedade que torna o sync robusto a falhas de rede.

O cenário sem idempotência

Capataz registra pesagem · clica sync · servidor recebe e insere · resposta se perde no caminho.

Cliente acha que falhou · tenta de novo · servidor insere de novo.

Boletim sai com 2 pesagens para o mesmo bovino no mesmo dia.

O cenário com idempotência

Mesma sequência, mas client_uuid é PK no servidor.

Segundo INSERT: changes = 0. Vira skipped na resposta.

Cliente confia: "o servidor já tem isso". Boletim mostra 1 pesagem.

-- schema do servidor
CREATE TABLE evento_recebido (
  client_uuid  TEXT PRIMARY KEY,         -- a mágica
  tipo         TEXT NOT NULL,
  data         TEXT NOT NULL,
  capataz_id   INTEGER NOT NULL,
  retiro_id    INTEGER NOT NULL,
  bovino_id    INTEGER NOT NULL,
  observacao   TEXT,
  received_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
-- a operação:
INSERT OR IGNORE INTO evento_recebido (...) VALUES (...);

Backend Node · 60 linhas de Express

better-sqlite3 · CORS · um statement por endpoint · uma transação por payload.

const express  = require('express');
const Database = require('better-sqlite3');
const cors     = require('cors');

const db = new Database('brpec-server.db');
db.pragma('journal_mode = WAL');
db.exec(`CREATE TABLE IF NOT EXISTS evento_recebido ( client_uuid TEXT PRIMARY KEY, ... );`);

const stmt = db.prepare(
  `INSERT OR IGNORE INTO evento_recebido (client_uuid, tipo, data, capataz_id, retiro_id, bovino_id, observacao)
      VALUES (@client_uuid, @tipo, @data, @capataz_id, @retiro_id, @bovino_id, @observacao);`
);

const app = express();
app.use(cors());
app.use(express.json());

app.post('/sync/deltas', (req, res) => {
  const deltas = req.body?.deltas;
  if (!Array.isArray(deltas)) return res.status(400).json({ ok: false });

  const tx = db.transaction((items) => {
    const accepted_uuids = []; let skipped = 0;
    for (const d of items) {
      const info = stmt.run(d);
      if (info.changes === 1) accepted_uuids.push(d.client_uuid);
      else                    skipped++;
    }
    return { accepted: accepted_uuids.length, skipped, accepted_uuids };
  });

  res.json({ ok: true, ...tx(deltas) });
});

app.listen(3000);

Por que síncrono (better-sqlite3 vs sqlite3)? 30k inserts/s, sem callback hell, leitura linear — ideal para a turma ler.

"Todo código nasce de um teste que falha" · spec mínimo

Mantemos a cultura de TDD, mas com 1 teste-âncora — o contrato de idempotência.

// server.test.js
const test   = require('node:test');
const assert = require('node:assert/strict');
const request = require('supertest');
const { createServer } = require('./server');

test('reenviar não duplica (idempotência)', async () => {
  const { app, close } = createServer({ dbPath: ':memory:' });

  const delta = {
    client_uuid: 'fixed-A', tipo: 'pesagem',
    data: '2026-05-14',
    capataz_id: 1, retiro_id: 1, bovino_id: 1
  };

  const r1 = await request(app).post('/sync/deltas').send({ deltas: [delta] });
  assert.equal(r1.body.accepted, 1);
  assert.equal(r1.body.skipped,  0);

  const r2 = await request(app).post('/sync/deltas').send({ deltas: [delta] });
  assert.equal(r2.body.accepted, 0);
  assert.equal(r2.body.skipped,  1);

  close();
});

Por que esse teste e não outros?

Idempotência é o contrato pedagógico da aula. É o que separa "sync ingênuo" de "sync confiável".

Outros testes (validação de campo, snapshot vazio, etc.) viram material de leitura no server.test.js entregue.

Cultura TDD herdada do Afonso

  • RED → GREEN → REFACTOR continua sendo o ciclo.
  • "Celebrar o vermelho" — o primeiro npm test falhando é vitória.
  • Aprofundamento de TDD em camadas → aula 10 (Testes & Automação).

Pergunta 1 de 3 · UUID no cliente

Por que o client_uuid é gerado no celular do capataz, e não no servidor quando ele recebe o evento?

⏱ 30 segundos · pense no capataz a 4 horas do escritório

Resposta-âncora

Porque, no momento da coleta, o servidor pode estar inacessível — e o evento precisa de uma identidade estável mesmo assim. Três consequências práticas:

① Idempotência: com id estável, reenvios viram no-op naturais. Sem id estável, o servidor não tem como reconhecer "isso já chegou" — daí duplicatas.
② Auditoria local: o capataz consegue rastrear "eu registrei isso" sem depender do servidor confirmar.
③ Sync de dois sentidos: quando o servidor mandar um snapshot de volta ("aqui está o que sei dos seus eventos"), o cliente reconhece pelos UUIDs e não cria duplicatas locais.

BrPec real: o capataz pesa 12 bovinos no retiro Cocho Novo na quinta-feira; volta ao escritório na segunda; sincroniza; rede cai no meio; reenvia. Sem UUID no cliente: 24 pesagens. Com UUID: 12.

Pergunta 2 de 3 · offline-first vs offline-only

A PWA do BrPec é "offline-first" ou "offline-only"? Qual a diferença, e por que escolhemos a primeira?

⏱ 30 segundos

Resposta-âncora

Offline-only = o app não conhece servidor. Tudo mora no celular, para sempre. Bom para to-do lists e diários pessoais; péssimo para um ecossistema de fazenda com 1 coordenador + N capatazes.

Offline-first = o app funciona offline (todas as operações principais), mas sabe sincronizar quando online. O caminho online é uma evolução, não uma regressão.

Escolhemos offline-first porque:

① Realidade do retiro: WiFi raro ou nulo. Não dá pra perder coleta esperando sinal.
② Necessidade do coordenador: dashboard consolidado dos N capatazes. Sem servidor, não tem dashboard.
③ Auditoria: o SGBD da fazenda é a fonte da verdade, não cada celular individual.

Diferença sutil que importa: em offline-first, "estar online" é uma otimização; em offline-only, é uma limitação. Mude o framing e o produto muda.

Pergunta 3 de 3 · service worker vs app

Por que o service worker vive em um arquivo separado da página (sw.js) e roda em uma thread diferente?

⏱ 30 segundos · pense no ciclo de vida do app vs do SW

Resposta-âncora

Três razões — cada uma resolveria um problema sozinha, mas as três juntas definem o que SW é:

① Ciclo de vida independente. O SW continua existindo depois que a aba fecha. Pode receber push notification, sincronizar em background, lidar com fetch. Se estivesse acoplado à página, tudo morreria com ela.
② Interceptação de requisições. O browser precisa de uma camada entre a página e a rede. O SW é essa camada. Se vivesse na página, intermediar a si mesmo geraria recursão e ciclos.
③ Thread separada (não bloqueia a UI). Cache, IndexedDB, criptografia podem ser pesados. Numa thread isolada, a UI segue fluida mesmo com SW trabalhando.

Consequência prática: você atualiza a página, mas o SW antigo continua servindo o cache antigo até skipWaiting() + clients.claim(). É por isso que mudanças em PWA às vezes "demoram" — é o SW antigo ainda no comando. brpec-pwa-v1 → brpec-pwa-v2 força a virada.
