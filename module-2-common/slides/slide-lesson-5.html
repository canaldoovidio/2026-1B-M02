O que é Node.js?

JavaScript fora do navegador. Runtime baseado em V8 (mesma engine do Chrome) + APIs de sistema (arquivos, rede, processos).

3 ideias-âncora

  • Event loop. Um único thread principal cuida de I/O assíncrono — perfeito para servidores web (muitas conexões, pouca CPU por requisição).
  • npm. O maior ecossistema de pacotes do mundo. Express, better-sqlite3, supertest — todos viram npm install e require().
  • Mesma linguagem dos dois lados. O front (PWA da aula passada) e o back (de hoje) compartilham JS. Models e validações podem ser literalmente o mesmo arquivo.

Comparativo rápido: Python tem Flask/FastAPI; Java tem Spring; Node tem Express. A diferença não é "qual é melhor" — é onde o ecossistema do parceiro vai. No BrPec, JS+Node casa porque a PWA já é JS.

SWEBOK v4 · Cap. 2 · separar estrutura e comportamento

A literatura por trás de "Models e Controllers". Sem isso, MVC vira jargão sem ancoragem.

SWEBOK v4 · Structural and Behavioral Design

IEEE Computer Society · 2024 · Capítulo 2 do Knowledge Area "Software Design"

Structural design — como o sistema é dividido em partes: módulos, camadas, interfaces, dependências.

Behavioral design — como as partes colaboram em tempo de execução: fluxos, eventos, casos de uso.

A regra: defina os limites antes de definir os fluxos. Cada parte deve ter um motivo para mudar (single responsibility).

Por que isso importa agora?

MVC é uma escolha estrutural (3 camadas, 3 responsabilidades). Os controllers definem o comportamento (o que cada rota faz). Sem essa separação, o servidor vira "um arquivo gigante chamado app.js" — funciona, mas não evolui.

  • Model = estado + invariantes (o capataz tem nome, RG, retiro_id; nunca nasce sem retiro).
  • View = como o estado vira saída (no backend HTTP, é o JSON que volta).
  • Controller = quem recebe a requisição, valida, chama o model, decide a resposta.

MVC no BrPec · uma responsabilidade por arquivo

Como o servidor de hoje se desdobra. O esqueleto pequeno; o padrão claro.

Model · evento.model.js

Define o que é um evento e como ele toca o banco.

  • Schema da tabela evento_recebido.
  • Statements preparados (INSERT OR IGNORE, SELECT).
  • Funções insertMany(deltas), findRecent(n).
  • Nunca conhece HTTP. Não sabe que existe um req/res.

Controller · sync.controller.js

Recebe a requisição, valida, orquestra o model, monta a resposta.

  • postDeltas(req,res) — valida payload, chama insertMany, devolve JSON.
  • getSnapshot(req,res) — chama findRecent, devolve JSON.
  • Nunca faz SQL direto. Não sabe se o banco é SQLite ou Postgres.

View / Routes · server.js

O "router" liga URL → controller. No HTTP, a "view" é o JSON de resposta.

  • app.post('/sync/deltas', postDeltas)
  • app.get('/sync/snapshot', getSnapshot)
  • Middlewares: cors(), express.json().

Conexão com a aula 4: o Model usa exatamente as mesmas tabelas e os mesmos JOINs que vocês aprenderam ontem — só que agora o motor SQLite roda no servidor Node, não no navegador. O SQL é o mesmo; muda quem digita.

Service Worker · ciclo de vida + estratégias de cache

3 eventos no SW. 2 estratégias. Tudo cabe num único arquivo sw.js.

1

install

Pre-cache da shell (HTML, CSS, JS, WASM, ícones, seed do banco).

→
2

activate

Limpa caches antigos. clients.claim() assume as abas abertas.

→
3

fetch

Intercepta cada requisição. Aqui mora a estratégia: cache-first ou network-first.

cache-first · para a shell

Tenta o cache. Se achou, devolve. Senão, busca rede e cacheia. Bom para HTML, CSS, JS, WASM, ícones, seed brpec.db.

network-first · para dados

Tenta a rede primeiro. Se falhou, devolve cache (ou JSON 503 estruturado). Bom para /sync/* — quer fresh, com degradação graciosa.

O SW do BrPec faz exatamente isso: SHELL_ASSETS em cache-first; /sync/* em network-first com fallback {ok:false, reason:'offline'}.

sql.js + IndexedDB · banco persistente no navegador

Evolução do sandbox da aula 4: o blob do banco vive no IndexedDB. Não some no reload.

Fluxo de boot (3 passos)

  1. Motor — initSqlJs() carrega o WASM.
  2. Blob — tenta ler do IDB (key brpec.db); se vazio, fetch o seed e grava no IDB.
  3. Banco — new SQL.Database(blob) abre em memória.

Após cada mutação: db.export() serializa → grava no IDB. Próximo boot reabre exatamente onde parou.

// db.js — núcleo da persistência
async function initDb() {
  const SQL = await initSqlJs({
    locateFile: (f) => `cdn/${f}`
  });

  let blob = await idbGet('brpec.db');
  if (!blob) {
    blob = await fetchSeed();
    await idbSet('brpec.db', blob);
  }

  const db = new SQL.Database(blob);
  async function persist() {
    await idbSet('brpec.db', db.export());
  }
  return { db, persist };
}

Por que IndexedDB e não OPFS? OPFS exige crossOriginIsolated (COOP/COEP), que GitHub Pages não serve. IndexedDB funciona em qualquer host. Portabilidade ganha.

UUID v4 · gerado no celular do capataz

Por que não deixar o servidor gerar o id? Porque o servidor pode estar a 4 horas de distância.

O contrato-chave

Cada evento, no momento da coleta, recebe um client_uuid gerado localmente. Ele acompanha o evento toda vez que ele atravessa a rede.

  • Sem internet, o registro já tem id estável.
  • Reenvios usam o mesmo id — servidor reconhece e ignora.
  • Auditoria liga o registro do celular ao registro do SGBD.

Geração: crypto.randomUUID() em contextos seguros; fallback crypto.getRandomValues em qualquer outro.

// portable UUID v4
function makeUUID() {
  if (crypto.randomUUID) return crypto.randomUUID();

  const b = new Uint8Array(16);
  crypto.getRandomValues(b);
  b[6] = (b[6] & 0x0f) | 0x40;
  b[8] = (b[8] & 0x3f) | 0x80;

  const h = [...b].map((x) => x.toString(16).padStart(2, '0'));
  return `${h[0]}${h[1]}${h[2]}${h[3]}-${h[4]}${h[5]}-${h[6]}${h[7]}-${h[8]}${h[9]}-${h.slice(10).join('')}`;
}

Backend Node · Models + Controller + Routes

O mesmo SQLite da aula 4, agora do outro lado da rede. Síncrono, simples, < 80 linhas.

// evento.model.js · MODEL
const Database = require('better-sqlite3');
const db = new Database('brpec-server.db');
db.pragma('journal_mode = WAL');
db.exec(`CREATE TABLE IF NOT EXISTS evento_recebido ( client_uuid TEXT PRIMARY KEY, tipo TEXT, data TEXT, capataz_id INT, retiro_id INT, bovino_id INT, observacao TEXT );`);
const stmtInsert = db.prepare(`INSERT OR IGNORE INTO evento_recebido (client_uuid,tipo,data,capataz_id,retiro_id,bovino_id,observacao) VALUES (@client_uuid,@tipo,@data,@capataz_id,@retiro_id,@bovino_id,@observacao);`);
function insertMany(deltas) { return db.transaction((items) => items.map((d) => stmtInsert.run(d).changes))(deltas); }
module.exports = { insertMany };

// sync.controller.js · CONTROLLER
const { insertMany } = require('./evento.model');
function postDeltas(req, res) {
  const deltas = req.body?.deltas;
  if (!Array.isArray(deltas)) return res.status(400).json({ ok: false });
  const changes = insertMany(deltas);
  const accepted = changes.filter((c) => c === 1).length;
  res.json({ ok: true, accepted, skipped: deltas.length - accepted });
}
module.exports = { postDeltas };

// server.js · ROUTES
const express = require('express');
const cors = require('cors');
const { postDeltas } = require('./sync.controller');
const app = express();
app.use(cors()); app.use(express.json());
app.post('/sync/deltas', postDeltas);
app.listen(3000);

Por que síncrono (better-sqlite3 vs sqlite3 async)? 30k inserts/s, sem callback hell, leitura linear — ideal para a turma ler. Express absorve a I/O do HTTP; o SQL fica simples.

Sync + Idempotência · "fazer X duas vezes vira fazer X uma vez"

A propriedade que torna o sync robusto a falhas de rede.

Sem idempotência

Capataz registra pesagem · clica sync · servidor recebe e insere · resposta se perde no caminho.

Cliente acha que falhou · tenta de novo · servidor insere de novo.

Boletim sai com 2 pesagens para o mesmo bovino no mesmo dia.

Com idempotência

Mesma sequência, mas client_uuid é PK no servidor.

Segundo INSERT OR IGNORE: changes = 0. Vira skipped na resposta.

Cliente confia: "o servidor já tem isso". Boletim mostra 1 pesagem.

-- contrato do servidor (no Model)
CREATE TABLE evento_recebido (
  client_uuid  TEXT PRIMARY KEY,         -- a mágica · veio do cliente
  tipo         TEXT NOT NULL,
  data         TEXT NOT NULL,
  capataz_id   INTEGER NOT NULL,
  retiro_id    INTEGER NOT NULL,
  bovino_id    INTEGER NOT NULL,
  observacao   TEXT,
  received_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

Test-Driven Development · Red → Green → Refactor

Antes do código, o teste. O teste é o contrato; a implementação é só uma forma de cumpri-lo.

🔴 RED

Escrever o teste que falha. Ele descreve o comportamento desejado.

→

🟢 GREEN

Mínimo de código para passar. Não resolva problemas futuros.

→

🔵 REFACTOR

Reorganize sem mudar comportamento. Teste verde garante a rede.

↻
// server.test.js · 1º teste-âncora
const test   = require('node:test');
const assert = require('node:assert/strict');
const request = require('supertest');
const { createServer } = require('./server');

test('reenviar não duplica (idempotência)', async () => {
  const { app, close } = createServer({ dbPath: ':memory:' });
  const delta = { client_uuid: 'fixed-A', tipo: 'pesagem', data: '2026-05-14', capataz_id: 1, retiro_id: 1, bovino_id: 1 };

  const r1 = await request(app).post('/sync/deltas').send({ deltas: [delta] });
  assert.equal(r1.body.accepted, 1);

  const r2 = await request(app).post('/sync/deltas').send({ deltas: [delta] });
  assert.equal(r2.body.accepted, 0);
  assert.equal(r2.body.skipped,  1);
  close();
});

Por que esse teste?

Idempotência é o contrato pedagógico da aula. Separa "sync ingênuo" de "sync confiável". Outros testes (validação, snapshot vazio) ficam como leitura no server.test.js entregue.

Onde o TDD travou — e o que vem depois (SDD)

2 leituras provocadoras: por que TDD nem sempre funciona, e o que está nascendo no lugar.

🔧 TDD, Where Did It All Go Wrong

Ian Cooper · DevTernity · 2017 · ~1h vídeo

Tese: a maioria das equipes faz TDD errado — testa implementação em vez de comportamento. Resultado: suíte frágil, refactor caro, código preso ao teste.

O que mudar: escrever testes contra portas (entrada do sistema), não contra internals. Cada bug vira 1 teste novo, mas só na borda.

No BrPec: nosso teste de idempotência ataca a porta HTTP (POST + body + status), não o Model interno. Isso é TDD bem feito.

📐 Spec-Driven Development

Kiro (AWS · 2024) · spec-kit (GitHub · 2025) · Tessl (2025)

Tese: a próxima fronteira não é "escrever testes antes" — é declarar a especificação e deixar a IA gerar (1) código, (2) testes, (3) docs em conjunto.

3 ferramentas para conhecer: Kiro (IDE da AWS com spec-mode), spec-kit (CLI open-source do GitHub), Tessl (plataforma SaaS de specs executáveis).

No BrPec: o nosso spec ainda é texto + 1 teste-âncora. Em 2 anos, será brpec.spec.md gerando server.js + tests.

A mensagem da aula: testes não são burocracia. São o contrato executável que sobrevive a refactors, mudanças de stack, e até a substituições do autor original do código.

Pergunta 1 de 3 · UUID no cliente

Por que o client_uuid é gerado no celular do capataz, e não no servidor Node quando ele recebe o evento?

⏱ 30 segundos · pense no capataz a 4 horas do escritório

Resposta-âncora

Porque no momento da coleta o servidor pode estar inacessível — e o evento precisa de uma identidade estável mesmo assim. Três consequências:

① Idempotência: com id estável, reenvios viram no-op naturais no servidor (INSERT OR IGNORE).
② Auditoria local: o capataz rastreia "eu registrei isso" sem depender do servidor.
③ Sync bidirecional: quando o servidor mandar snapshot de volta, o cliente reconhece pelos UUIDs e não duplica localmente.

BrPec real: o capataz pesa 12 bovinos no retiro Cocho Novo; sincroniza; rede cai no meio; reenvia. Sem UUID no cliente: 24. Com UUID: 12.

Pergunta 2 de 3 · MVC e responsabilidades

Por que o Model não deve conhecer req e res? E por que o Controller não deve escrever SQL?

⏱ 30 segundos · pense na SWEBOK v4 cap. 2

Resposta-âncora

Porque cada camada tem um motivo para mudar (single responsibility — SWEBOK v4, structural design).

① Model conhece req/res → quando trocar de HTTP para fila (RabbitMQ, Kafka), tudo quebra. Model fala SQL e regras de negócio, e só.
② Controller escreve SQL → quando trocar SQLite por Postgres (ou MongoDB), tudo quebra. Controller orquestra, valida, formata resposta.
③ Resultado prático: dá pra testar Model sem HTTP, Controller sem banco real (mockar Model).

BrPec real: a aula 6 vai adicionar endpoints de leitura. Se respeitarmos MVC, basta criar evento.controller.js novo e reusar o Model. Se misturarmos, vai precisar reescrever tudo.

Pergunta 3 de 3 · TDD bem feito

Por que nosso teste de idempotência usa supertest + POST em vez de chamar insertMany() direto do Model?

⏱ 30 segundos · pense na crítica do Ian Cooper

Resposta-âncora

Porque queremos testar comportamento, não implementação. Atacando a porta HTTP (POST + body + status), o teste:

① Sobrevive a refactor. Se amanhã quebrarmos insertMany em duas funções, o teste continua verde.
② Documenta o contrato real. Quem lê o teste sabe o que o cliente HTTP vai ver.
② Pega bugs de integração. Falha de CORS, de middleware, de parsing JSON — tudo dentro do raio do teste.

É exatamente o que Ian Cooper defende em "TDD, Where Did It All Go Wrong": teste a porta, não as entranhas. Isso é TDD bem feito.
