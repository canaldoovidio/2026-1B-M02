🎯 Objetivo da Aula 10

Sair com o repositório verde no CI: cada camada coberta pelo tipo certo de teste, hooks impedindo merge ruim e GitHub Actions rodando a cada push.

Por que testar?
Quatro razões para investir tempo escrevendo código que valida outro código

🛡️ Regressão

Mudou o UserService e o login parou? Os testes falham antes do usuário falar. Cada bug consertado vira um teste — ele não volta sem ser detectado.

♻️ Refactoring seguro

Renomear, extrair função, trocar implementação do repositório — tudo possível quando há rede de segurança. Sem testes, refactor é apostar.

📖 Documentação executável

Ler OrderService.test.ts mostra como o código deve se comportar. Documentação que mente é detectada pelo CI; teste que mente é detectado pelo CI também.

🚀 Confiança para deployar

Deploy de sexta às 17h sem medo. O CI verde é a garantia objetiva de que, no mínimo, nada que estava coberto quebrou. O resto é sorte; coberto é certeza.

A Pirâmide de Testes
Muitos rápidos na base · Poucos lentos no topo
🌐 E2E
Poucos · lentos · frágeis. Subir o app inteiro + browser. Playwright. Cobre fluxos críticos de ponta a ponta.
🔗 Integração
Médios · banco real efêmero. Repositórios + controllers via supertest. Validam contratos entre camadas.
⚡ Unitários
Muitos · rápidos · isolados. Helpers e services com mocks. Centenas rodam em poucos segundos.

❌ Anti-padrão: Ice-Cream Cone

Pirâmide invertida — só E2E. Suite lenta, instável, ninguém roda local. Bugs descobertos no final do pipeline.

❌ Anti-padrão: Hourglass

Muitos unitários e muitos E2E, sem integração no meio. Camadas casam isoladamente mas falham juntas em produção.

Estrutura AAA e Naming
Arrange · Act · Assert — o esqueleto de todo bom teste
user.service.test.ts

🧱 As 3 fases

  • Arrange — prepara dados, mocks, estado inicial
  • Act — chama a função sob teste, uma vez
  • Assert — verifica saída e efeitos colaterais

📝 Naming descritivo

describe('UserService') agrupa por unidade. it('cria usuário com email único') descreve o comportamento — não a implementação. Ler a saída do test runner deve parecer um texto.

Helpers — testes triviais (1:1)
Funções puras: entrada → saída · sem side-effect
helpers/format.ts & .test.ts

✨ Por que helpers são fáceis

  • Sem banco, sem rede, sem framework
  • Mesma entrada → mesma saída sempre
  • Edge cases em 3 linhas: zero, negativo, NaN
  • Cada teste roda em < 1ms

🎯 Regra prática

Cada helper público (formatCurrency, slugify, isValidCpf) merece pelo menos 3 testes: caminho feliz, borda e entrada inválida.

Services — onde mora a lógica
Unitário com mock do Repository
order.service.test.ts

🎭 Test Doubles

  • Stub — devolve valor fixo
  • Spy — observa chamadas (real ou stub)
  • Mock — espera comportamento + verifica
  • Fake — implementação simplificada (in-memory)

🧠 No service, mocke o repo

O teste do service NÃO bate no banco. Ele garante apenas que "se o repo disser X, o service decide Y". Banco fica para o teste do repo.

Repositories — banco real efêmero
Aqui você precisa do Postgres de verdade
user.repository.test.ts

🚫 Não mocke o repo no próprio teste do repo

O propósito desse teste é validar que o SQL real funciona. Mockar o pg aqui é testar que o mock devolve o que você programou — inútil.

♻️ Estratégias de isolamento

  • Postgres no Docker ou Testcontainers
  • BEGIN + ROLLBACK por teste
  • TRUNCATE em beforeEach (mais simples)
  • Banco descartável por suite (CI)
Controllers — supertest no Express
Validar status, headers, JSON e fluxo HTTP
user.controller.test.ts

🌐 supertest = HTTP sem rede

Importa o app do Express e dispara requisições direto na função, sem subir servidor. Rápido, determinístico, fácil de assertar.

✅ O que validar

  • Status code (201, 404, 409)
  • Shape do JSON (expect(body).toMatchObject(...))
  • Headers (Location em POST, Set-Cookie)

🖼️ Views EJS — smoke test

Não se testa EJS direto. O controller renderiza, o supertest checa que <title>...</title> aparece. Visual fica para Playwright.

Coverage — métrica que mente sem você
100% de cobertura ≠ código correto
npm run test:cov

📊 As 4 dimensões

  • Statements — linhas executadas
  • Branches — galhos de if/else testados
  • Functions — funções chamadas
  • Lines — linhas físicas tocadas

⚠️ Coverage não garante

Linha tocada sem assert não testa nada. Você pode ter 100% e zero garantia. Use coverage para achar buracos, nunca como meta. Meta: equipe escreve teste antes do bug.

TDD — Red, Green, Refactor
O ciclo curto que muda como você escreve código
🔴
RED
Escreva o teste primeiro. Ele falha — porque o código nem existe ainda.
→
🟢
GREEN
Implemente o mínimo para o teste passar. Feio é OK.
→
🔧
REFACTOR
Melhore o desenho com a rede de segurança verde. Repita.

💡 O que muda na prática

Você só escreve código que tem teste. Não existe "depois eu testo". O design da API emerge das chamadas que o teste faz — fica naturalmente mais usável.

🎯 Quando vale a pena

Lógica de negócio, bugs reportados (escreva o teste que reproduz antes de corrigir), refactors arriscados. Para CRUD trivial, comece pelo controller.

Husky + lint-staged — pre-commit blindado
Lixo não entra na branch
.husky/pre-commit & package.json

🪝 Hooks que importam

  • pre-commit — eslint + prettier nos arquivos staged
  • pre-push — tsc --noEmit + jest rápido
  • commit-msg — commitlint (convenção)

⚡ lint-staged é a chave

Roda eslint só nos arquivos modificados. Commit em < 2s mesmo em projeto grande. Sem lint-staged, hook fica lento, dev pula com --no-verify, hook morre.

GitHub Actions — CI a cada push
Pipeline mínimo: install · lint · test · coverage
.github/workflows/ci.yml

🐳 Postgres como service

O job sobe um container Postgres ao lado dos testes. DATABASE_URL aponta para localhost:5432 — os testes de repository conectam normalmente.

📊 Cobertura como artefato

Suba coverage/lcov.info com actions/upload-artifact para ver o relatório no PR. Bonus: integrar Codecov.

✅ Boas práticas

  • Determinismo: sem Math.random() sem seed
  • Independência: ordem dos testes não importa
  • Velocidade: unitário < 50ms cada
