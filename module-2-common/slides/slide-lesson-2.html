O que é um Banco de Dados?
Por que não basta uma planilha?

📁 O que é?

Coleção organizada de dados persistentes, estruturados de forma que possam ser acessados, gerenciados e atualizados com eficiência.

🤔 Por que precisamos?

  • Múltiplos usuários simultâneos
  • Integridade e consistência dos dados
  • Consultas complexas e rápidas
  • Persistência, backup e recuperação

Instagram (posts/likes), Spotify (playlists), iFood (pedidos), sistemas bancários, cadastros de alunos no Inteli.

🔐 Gerencia acesso

✅ Garante integridade

Constraints (PK, FK, NOT NULL, UNIQUE) impedem que dados inválidos entrem no banco.

🟦 Entidade

Substantivo — algo com existência própria no domínio do sistema. Ex.: USUARIO, PEDIDO, PRODUTO

🔷 Atributo

Característica de uma entidade. Ex.: nome, email, preço. O atributo id com PK identifica cada linha unicamente.

🔗 Relacionamento

Verbo entre entidades. "USUARIO faz PEDIDO" · "PEDIDO contém PRODUTO". Implementado com Foreign Keys.

Diagrama ER — Notação de Chen
Quadrado, losango e oval — e a regra do N:N

🔣 Símbolos da notação

Quadrado = entidade (substantivo) · Losango = relacionamento (verbo) · Oval = atributo. Linhas conectam com cardinalidade 1, N, M.

🔁 Regra do N:N — quadrado em volta do losango

Quando o relacionamento é muitos-para-muitos, ele vira uma tabela associativa com PK própria — e pode carregar atributos como quantidade.

Modelo Lógico — Da ER às Tabelas

📐 Regras de tradução

Entidade → tabela · Atributo → coluna · 1:N → FK no lado muitos (pedidos.usuario_id → usuarios.id) · N:N → tabela associativa com PK composta.

🔁 Tabela associativa pedido_produto

Nasce do losango contém com quadrado em volta. Tem PK composta (pedido_id + produto_id) — ambas FKs — e absorve o atributo quantidade do relacionamento.

Tipos de Dados SQL
Escolha o tipo certo para cada coluna
INTEGER / SERIAL
Números inteiros
SERIAL é auto-increment — ideal para chaves primárias.
id SERIAL PRIMARY KEY
VARCHAR(n) / TEXT
Textos
VARCHAR limita o tamanho. TEXT não tem limite definido.
nome VARCHAR(100)
DECIMAL(p,s)
Decimais precisos
p = total de dígitos, s = casas decimais. Use para dinheiro.
preco DECIMAL(10,2)
BOOLEAN
Verdadeiro / Falso
Aceita true/false ou 1/0. Útil para flags de estado.
ativo BOOLEAN DEFAULT true
TIMESTAMP / DATE
Data e hora
TIMESTAMP inclui hora. DATE é apenas data. NOW() retorna o instante atual.
criado_em TIMESTAMP
JSON / JSONB
Dados semi-estruturados
JSONB é indexável e mais rápido. Útil para metadados flexíveis.
config JSONB
Constraints — Regras de Integridade
O banco garante que só dados válidos entram
PRIMARY KEY
Identifica unicamente cada linha
NOT NULL
Campo obrigatório
UNIQUE
Valor não pode repetir
FOREIGN KEY
Liga tabelas, garante integridade referencial
DEFAULT
Valor padrão quando não informado

💡 Por que constraints importam?

Elas garantem que a aplicação não precise validar tudo manualmente. O banco rejeita automaticamente dados inválidos, protegendo a integridade mesmo com bugs no código.

DML — INSERT INTO
Data Manipulation Language · Inserindo registros na tabela
dml_insert.sql — PostgreSQL
INSERT básico
Linha única
Especifica colunas explicitamente. Colunas com DEFAULT e SERIAL podem ser omitidas.
INSERT múltiplo
Várias linhas de uma vez
Muito mais eficiente que múltiplos INSERTs individuais. Um único round-trip ao banco.
INSERT … RETURNING
Retorna o ID gerado
Elimina a necessidade de um SELECT após o INSERT para buscar o id autogerado.
DML — UPDATE e DELETE
Data Manipulation Language · Alterando e removendo registros
dml_update_delete.sql
⚠️ UPDATE sem WHERE
Atualiza TODOS os registros
UPDATE usuarios SET ativo = false
Sem WHERE → todas as linhas são alteradas. Sempre use WHERE.
🚨 DELETE sem WHERE
Apaga TODOS os registros
A tabela fica vazia mas continua existindo. Use TRUNCATE para isso intencionalmente. Sempre use WHERE no DELETE.
✅ Padrão seguro
Filtre sempre pelo ID
Use WHERE id = $1 em aplicações. Nunca confie em input direto do usuário — use prepared statements.
DCL + TCL — Controle e Transações
Permissões de acesso e operações ACID
dcl_tcl.sql — PostgreSQL

🔐 DCL — Data Control Language

GRANT: concede permissões a um usuário ou role
REVOKE: remove permissões
Permissões granulares: SELECT, INSERT, UPDATE, DELETE, ALL

🔄 TCL — Transaction Control Language

BEGIN: inicia uma transação
COMMIT: confirma permanentemente todas as operações
ROLLBACK: desfaz tudo desde o BEGIN
Garante as propriedades ACID

⚡ ACID

Atomicity · Consistency · Isolation · Durability

🐘 PostgreSQL
Open-source · padrão SQL avançado · 1996
postgres.sql — PostgreSQL 16

🔢 Auto-increment

SERIAL cria uma sequence automática. Em versões novas, GENERATED AS IDENTITY é o padrão SQL.

📦 Tipos avançados

JSONB indexável, ARRAY, UUID, INET, tipos geográficos com PostGIS.

🎯 Quando usar

SaaS modernos, dados complexos, máxima conformidade com SQL padrão.

🐬 MySQL / MariaDB
O mais popular da web · LAMP, WordPress · 1995
mysql.sql — MySQL 8

🔢 Auto-increment

AUTO_INCREMENT direto na coluna. Não precisa de sequence separada.

⚙️ Storage Engine

ENGINE=InnoDB suporta FK e transações. MyISAM não suporta — escolha sempre InnoDB.

🎯 Quando usar

Web apps clássicos, hospedagem compartilhada, ecossistema PHP/WordPress.

💾 SQLite
Embedded · zero-config · um único arquivo · 2000
app.db — SQLite 3

🔢 Auto-increment

INTEGER PRIMARY KEY já é alias do rowid. AUTOINCREMENT só evita reuso de IDs.

🌀 Tipagem dinâmica

Apenas 5 storage classes: NULL, INTEGER, REAL, TEXT, BLOB. VARCHAR(100) é ignorado.

🎯 Quando usar

Apps mobile, protótipos, testes, banco embarcado em cada instalação.

🟥 SQL Server (T-SQL)
Microsoft · stack .NET · enterprise · 1989
tsql.sql — SQL Server 2022

🔢 Auto-increment

IDENTITY(seed, increment) — IDENTITY(1,1) começa em 1 e soma 1 a cada linha.

🔤 NVARCHAR vs VARCHAR

NVARCHAR guarda Unicode (acentos, emojis); VARCHAR só ASCII estendido. Use NVARCHAR sempre.

🎯 Quando usar

Empresas Microsoft, integrações com Power BI / Azure, BI corporativo.

🔴 Oracle Database
Enterprise · grandes corporações · bancos / telecom · 1979
oracle.sql — Oracle 23ai

🔢 Auto-increment

GENERATED ALWAYS AS IDENTITY (12c+). Antes era SEQUENCE + TRIGGER.

🔤 Tipos próprios

VARCHAR2 em vez de VARCHAR, NUMBER(p,s) universal, não tem BOOLEAN — usa NUMBER(1).

🎯 Quando usar

Sistemas legados, bancos, telecom, casos onde performance e suporte enterprise são críticos.

🚀 Mão na Massa — Crie sua Tabela no Supabase
Modele 1 tabela do problema do seu parceiro · ~10 min · pequena para caber na aula
📋 Passo a passo
  1. Acesse seu projeto em supabase.com
  2. Abra SQL Editor no menu lateral
  3. Identifique 1 entidade pequena do parceiro (ex.: cliente, agendamento, tarefa)
  4. Modele: 3–5 colunas + PK + 1 constraint útil
  5. Escreva o CREATE TABLE em PostgreSQL
  6. Execute com Ctrl+Enter (ou botão Run)
  7. Verifique em Table Editor que a tabela apareceu
💡 Exemplo de referência
Supabase SQL Editor — PostgreSQL

⏱️ Escopo

1 tabela só. 3–5 colunas. Não precisa modelar o domínio inteiro — escolha algo que caiba em 10 minutos.

🔗 Use o que aprendemos

SERIAL PRIMARY KEY · VARCHAR(N) NOT NULL · TIMESTAMP DEFAULT NOW() · UNIQUE.

✅ Critério

Tabela criada com sucesso e visível no Table Editor. Bonus: insira 1 linha com INSERT INTO.
