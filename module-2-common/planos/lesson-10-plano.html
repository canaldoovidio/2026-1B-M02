Resumo

A aula 10 traz garantia para tudo que foi construído nas aulas anteriores. Os alunos cobrem cada camada do MVC com o tipo de teste certo: helpers com unit puro, services com mock do repository, repositories com banco efêmero (pg-mem ou Postgres em Docker), controllers com supertest, e views EJS com smoke pelo controller.

Coverage entra como métrica (não meta). Husky + lint-staged travam erros no pre-commit. GitHub Actions roda lint + tests com Postgres como service container. A pirâmide de testes vira regra prática a partir desta aula.

Objetivos de aprendizagem

  • OA-1Posicionar cada teste na camada correta da pirâmide (unit / integração / e2e).
  • OA-2Aplicar AAA (Arrange/Act/Assert) com naming describe/it claro e legível.
  • OA-3Mockar o repository em testes de service usando jest.fn().
  • OA-4Configurar banco efêmero para repository (pg-mem, Testcontainers ou Postgres descartável).
  • OA-5Testar controllers com supertest cobrindo status, shape JSON e redirecionamentos.
  • OA-6Configurar Husky + lint-staged + GitHub Actions com Postgres como service.

Pré-requisitos

  • Aulas 5–9 concluídas: o projeto já existe com camadas implementadas (controllers, services, repositories, helpers e views EJS).
  • Camadas separadas em diretórios distintos (src/controllers, src/services, src/repositories, src/helpers).
  • Jest e ts-jest instalados no projeto (caso ainda não, instalar no Bloco 1).
Importante

Aqui a equipe cobre o que ficou descoberto — o trabalho não é "escrever testes do zero", é auditar o projeto e tapar buracos camada a camada.

TalkPirâmide de testes + anti-padrões

Apresentação da pirâmide (muitos unit, alguns integration, poucos e2e), o anti-padrão "casquinha de sorvete" (e2e demais) e o "ampulheta" (sem integração). Como cada camada do MVC mapeia para a pirâmide.

CodingService com mock de repository (red→green)

Demo: testar um service usando jest.fn() para fingir o repository, isolando a regra de negócio. Os pares replicam o padrão em um service do próprio projeto e debatem quando o mock está "demais".

Demonstração de duas estratégias: pg-mem (in-memory rápido) e Postgres em Docker (mais fiel). Padrão BEGIN/ROLLBACK por teste para garantir isolamento. Os pares escolhem uma estratégia e escrevem 1 teste de repository.

Em pares: importar o app Express, usar supertest para fazer request(app).post('/...').expect(201), validar status, shape JSON e redirecionamentos. Discussão: onde plugar o banco efêmero?

11h45
11h55

TalkCoverage como métrica + Husky + GitHub Actions

Coverage como métrica (não meta) — número alto sem assert significativo é teatro. Apresentação rápida do pre-commit com Husky + lint-staged e do workflow GitHub Actions com Postgres como service container.

11h55

AtivaSíntese + entrega da tarde

Cada aluno escreve "a aula 10 me ensinou que...". O professor anuncia a entrega: cobertura por camada + Husky + workflow verde no MR chore(test): cobertura por camada + CI.

Estratégias de metodologia ativa

  • Dojo de testes — cada par escreve 1 teste por camada e roda no projetor, expondo o ciclo red→green em tempo real.
  • Debate sobre over-mocking — após o exercício de service, a turma discute quando mockar protege e quando esconde bug real.
  • Code review cruzado — equipes trocam o ci.yml recém-escrito para validar steps, services e cache antes de abrir o MR.

Verificação de aprendizagem

  • CR-1Pelo menos 1 helper testado com AAA explícito (Arrange/Act/Assert) e naming describe/it claro.
  • CR-2Pelo menos 1 service testado mockando o repository com jest.fn().
  • CR-3Pelo menos 1 repository testado com banco efêmero (pg-mem, Testcontainers ou Postgres descartável) com isolamento por teste.
  • CR-4Pelo menos 1 controller testado com supertest cobrindo status e shape JSON.
  • CR-5Husky + lint-staged ativos no pre-commit + workflow GitHub Actions verde, tudo no MR chore(test): cobertura por camada + CI aberto até as 16h.
Critério mínimo

Aluno que sair às 16h sem pelo menos um teste por camada (helper, service, repository, controller) e sem o workflow do GitHub Actions verde está em débito técnico — o projeto não pode subir para produção sem rede de proteção mínima.

Conexão com a aula 11

A aula 11 fecha o módulo olhando para baixo da nossa stack: a infraestrutura de redes — TCP/IP, DNS, HTTPS+TLS, CDN, cache HTTP e padrões de tolerância a falhas — que sustenta tudo em produção.

  • Os controllers testados hoje passarão a receber requests cruzando CDN, cache e TLS — entender essa cadeia muda como projetamos endpoints.
  • Os testes que você escreveu serão os primeiros a quebrar quando rede falhar: timeouts, retries e idempotência viram parte do projeto na próxima aula.
Para o autoestudo da aula 11

Antes das 8h do próximo dia: rodar dig no domínio do projeto, ler o material da aula 11 e mapear quais respostas do back-end fazem sentido cachear (e por quanto tempo).
