Resumo

A aula 5 abre o bloco de back-end com uma motivação concreta: o capataz do BrPec opera no campo, sem WiFi. Em vez de tratar isso como limitação, transformamos em arquitetura — uma Progressive Web App que funciona offline e sincroniza quando a rede voltar.

Tecnicamente, fazemos quatro coisas em sequência: (1) damos persistência ao banco no navegador (sql.js export → IndexedDB → reload), (2) tornamos a página instalável (manifest + service worker com cache-first), (3) geramos UUID v4 no cliente como identidade estável do registro e (4) escrevemos um servidor Node com INSERT OR IGNORE que reconcilia idempotentemente.

O espírito da cultura TDD do material original do Afonso continua presente — mas com 1 spec mínimo (idempotência via supertest). Aprofundamento em testes em camadas vai para a aula 10 (Testes & Automação).

Fechamos com uma ponderada de 30 min em sala: cada aluno entrega o print do Lighthouse PWA score (≥ 80) da sua versão funcionando local + vídeo curto + ADR de 3 frases.

Objetivos de aprendizagem

  • OA-1Identificar os 3 pilares de uma PWA (HTTPS / manifest / service worker) e relacionar cada um a sua função.
  • OA-2Escrever um manifest.webmanifest mínimo e funcional, com ícone, theme color e display:standalone.
  • OA-3Implementar um service worker que cacheia a shell (cache-first) e degrada graciosamente endpoints de dados (network-first com 503 estruturado).
  • OA-4Persistir o blob do banco SQLite (sql.js) em IndexedDB, recuperando o estado entre sessões.
  • OA-5Gerar UUID v4 no cliente (com fallback para contexto não-seguro) e justificar a decisão pedagógica.
  • OA-6Construir um servidor Express + better-sqlite3 com POST /sync/deltas idempotente via INSERT OR IGNORE.
  • OA-7Escrever e rodar um spec de idempotência com node --test + supertest.
  • OA-8Avaliar a própria PWA com o Lighthouse e atingir score ≥ 80 na categoria PWA.

Detalhamento da instrução (10:00 — 11:15)

Sequência completa do que rola em cada bloco, mapeado para os slides.

10:00 — 10:15 · Daily

  • Slide 3 com cronômetro. Pergunta projetada: "Que dado seu app precisaria no celular do capataz quando não tem internet?"
  • Cada aluno 1 min: "ele precisa LER X · ESCREVER Y · SINCRONIZAR Z".
  • Professor anota padrões no quadro: o mais comum vai ser "ler antigo, escrever novo" — que é exatamente o shape que a PWA otimiza.

10:15 — 10:30 · Recap + 3 pilares

  • Slide 2 — três limitações concretas do sandbox da aula 4.
  • Slide 4 — 3 pilares da PWA: HTTPS, manifest, service worker. Cada um é um dos 3 problemas resolvidos.
  • Slide 5 — manifest detalhado (campo a campo).

10:30 — 10:50 · Persistência + sync

  • Slide 6 — SW lifecycle (install/activate/fetch).
  • Slide 7 — cache strategies (cache-first vs network-first com exemplo do BrPec).
  • Slide 8 — sql.js + IndexedDB. Mostrar db.js da PWA real.
  • Slide 9 — UUID v4 no cliente (com fallback). Justificar pedagogicamente.
  • Slide 10 — diagrama PWA ↔ servidor + endpoints. Slide 11 — idempotência: cenário com vs sem.

10:50 — 11:05 · Servidor Node + spec

  • Slide 12 — server.js comentado (60 linhas).
  • Slide 13 — server.test.js. Rodar npm test ao vivo no terminal projetado: 5 testes verdes.
  • Reforçar: "celebrar o vermelho" → para a turma 26, o vermelho está implícito quando better-sqlite3 não está instalado. Faça rm -rf node_modules ao vivo, rode npm test, mostre o crash, instale, mostre verde. É RED → GREEN em 30 segundos.

11:05 — 11:15 · Demo end-to-end

  • Slide 14 — roteiro da demo. Abrir a PWA real, projetada.
  • Cortar rede no DevTools → registrar 3 eventos com bovinos diferentes → counter "pendentes: 3".
  • Religar rede → clicar sync → response {accepted: 3}.
  • Clicar sync de novo → {accepted: 0, skipped: 3}. Comentar: "isso aqui é a idempotência funcionando — sem ela, teríamos 6 eventos no servidor."
  • Abrir terminal do server: sqlite3 brpec-server.db "SELECT * FROM evento_recebido;" → confirmar.

Bônus universal

Qualquer grupo: abrir o Lighthouse (DevTools → Lighthouse → PWA) e capturar o score. É exatamente o que vão entregar na ponderada — quem fizer no lab já tem ⅓ pronto.

Ponderada em sala (11:30 — 12:00)

Entrega individual com 3 artefatos:

  1. Print do Lighthouse PWA score da PWA rodando local (≥ 80 pontos).
  2. Vídeo de 30 segundos mostrando: app instalado, registro offline (1 evento), sync, banco do servidor confirmando.
  3. ADR mínimo em aula5-ponderada-NOME.md: 3 frases respondendo "qual foi a decisão de design da minha PWA?" — pode ser sobre estratégia de cache, formato dos deltas, UI, etc.

Critérios de correção

  • CR-150% · Lighthouse PWA ≥ 80 (manifest válido + SW registrado + start_url cacheado + ícone).
  • CR-230% · Vídeo prova o ciclo completo (offline → registrar → sync → confirmação no servidor).
  • CR-315% · ADR coerente — escolha técnica explicitada com motivação.
  • CR-45% · Bônus: rodou os 5 specs do server (print do npm test verde anexo).
Critério mínimo

Quem entregar PWA que não chega a ≥ 50 no Lighthouse ou não consegue sincronizar 1 evento está em débito técnico para a aula 6 — onde o servidor ganha CRUD HTTP completo e isso amplia.

Estratégias de metodologia ativa

  • Lab cooperativo — os 5 grupos não fazem a mesma coisa; cada um faz um passo do ciclo. Juntar os prints monta a documentação completa em sala.
  • Predição antes da execução — antes de rodar o sync pela segunda vez, o professor pergunta: "quantos eventos vocês acham que vão entrar?" — força o aluno a articular o conceito de idempotência antes de ver.
  • Cultura TDD herdada — RED → GREEN visível ao vivo: rm -rf node_modules + npm test → crash; npm install → 5 verdes. Em 30 segundos.
  • Ponderada baseada em métrica externa — Lighthouse é uma ferramenta neutra, automática. Score < 80 não é "opinião do professor", é checklist objetivo do Google.
  • Aula 5 referencia aula 4 explicitamente — recapitulação não é repetição, é evolução: cada limitação do sandbox vira solução desta aula.

Conexão com a aula 6

A aula 6 (Back-End II — endpoints e documentação) expande o servidor de hoje:

  • De POST /sync/deltas + GET /sync/snapshot para CRUD HTTP completo (POST, PUT, DELETE).
  • Validação de payload com erros estruturados (400/422) — AppError + middleware.
  • Documentação automática (OpenAPI/Swagger ou alternativa minimalista).
  • Como a PWA consome esses endpoints quando online — refresh do snapshot, atualização local.
Para o autoestudo da aula 6

(1) Deixar o sync server rodando local. (2) Listar 3 endpoints que faltam para "fechar" o BrPec offline-first (sugestões: DELETE /sync/evento/:uuid, GET /relatorios/aftosa, PUT /sync/evento/:uuid). (3) Ler material aula 6 sobre validação + AppError.
