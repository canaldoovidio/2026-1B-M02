Resumo

A aula 5 abre o bloco de back-end com a pergunta certa: "se a PWA do capataz funciona offline, quem recebe os dados quando o sinal volta?" A resposta é um servidor Node.js bem-estruturado.

Tecnicamente, montamos o ciclo: (1) entender o que é Node.js — runtime JS no servidor, event loop, npm; (2) ler o Capítulo 2 do SWEBOK v4 (structural and behavioral design) como base teórica para Models e Controllers; (3) aplicar MVC ao BrPec — Model fala SQL, Controller orquestra HTTP, Routes ligam URL → controller; (4) revisitar a PWA de forma compacta (manifest + SW + sql.js + IndexedDB + UUID v4); (5) conectar tudo via POST /sync/deltas idempotente com INSERT OR IGNORE; (6) sustentar com TDD (node:test + supertest) atacando a porta do sistema, na linha do "TDD, Where Did It All Go Wrong" de Ian Cooper; (7) abrir janela para o futuro com Spec-Driven Development (Kiro, spec-kit, Tessl).

Esta aula não tem ponderada. O foco é a construção da stack inteira, com lab cooperativo de 40 minutos onde cada grupo executa um pedaço do fluxo PWA → Node → SQLite e juntam os artefatos no fim.

Objetivos de aprendizagem

  • OA-1Explicar o que é Node.js, o event loop e o papel do npm no ecossistema.
  • OA-2Aplicar a separação Models / Controllers / Routes (SWEBOK v4 Cap. 2) ao servidor BrPec.
  • OA-3Construir um servidor Express + better-sqlite3 com POST /sync/deltas idempotente.
  • OA-4Conectar a PWA da aula 4 ao servidor (HTTPS local), provando o ciclo offline → online.
  • OA-5Justificar por que o client_uuid é gerado no cliente — em termos de idempotência, auditoria e sync bidirecional.
  • OA-6Escrever e rodar um spec de idempotência com node --test + supertest atacando a porta HTTP.
  • OA-7Reconhecer a crítica de Ian Cooper (TDD WDIAGW): testar comportamento, não implementação.
  • OA-8Reconhecer Spec-Driven Development como evolução natural (Kiro, spec-kit, Tessl) — sem precisar implementar.

Cronograma do encontro

2 horas presenciais, divididas em 5 blocos: daily, exposição Node/MVC, revisita PWA, servidor + TDD, lab cooperativo.

Bloco 1 · Daily com pergunta de dado offline

10:00 — 10:15 · 15 min

"Que dado seu app precisa no celular do capataz quando ele não tem internet, e quem recebe esse dado quando o sinal voltar?" Cada aluno 1 min: ler / escrever / sincronizar. Cronômetro no slide.

Bloco 2 · Arquitetura + Node.js + MVC

10:15 — 10:40 · 25 min

(a) Slide-mestre da arquitetura PWA ↔ Node ↔ SQLite. (b) O que é Node.js (event loop, npm, JS dos dois lados). (c) SWEBOK v4 Cap. 2 — structural vs behavioral design. (d) MVC aplicado ao BrPec: Model (banco + regras), Controller (orquestra HTTP), Routes (URL → controller).

Bloco 3 · PWA · revisita compacta + sync

10:40 — 11:00 · 20 min

3 pilares (HTTPS + Manifest + SW), cache strategies (cache-first vs network-first), sql.js + IndexedDB (persistência), UUID v4 no cliente (idempotência), endpoint POST /sync/deltas com INSERT OR IGNORE.

Bloco 4 · Servidor Node + TDD + Spec-Driven

11:00 — 11:20 · 20 min

Anatomia do server.js + evento.model.js + sync.controller.js. Ciclo TDD (Red → Green → Refactor) com node:test + supertest — teste de porta, não de entranha (Ian Cooper). Janela para Spec-Driven Development (Kiro, spec-kit, Tessl).

Bloco 5 · Lab cooperativo nos 5 grupos

11:20 — 12:00 · 40 min · cronômetro

Cada grupo executa 1 etapa do ciclo PWA → Node → SQLite (subir servidor, instalar PWA, registrar offline, provar idempotência, verificar SQLite + rodar TDD). Juntando os 5 prints, a turma tem o ciclo completo documentado. Demo end-to-end no projetor antes do lab.

Sem ponderada hoje

A aula 5 desta turma não tem ponderada em sala. O foco é construção colaborativa da stack inteira e apropriação dos conceitos via lab cooperativo. A próxima avaliação ponderada virá na aula 6 ou conforme o cronograma do módulo.

Detalhamento da instrução (10:00 — 11:20)

Sequência do que rola em cada bloco, mapeada para os slides.

10:00 — 10:15 · Daily (slide 3)

  • Pergunta projetada: "Que dado precisa no celular sem internet, e quem recebe quando o sinal voltar?"
  • Cada aluno 1 min: "ler X · escrever Y · servidor Z recebe."
  • Professor anota padrões no quadro. A coluna "servidor Z" é a ponte para o Bloco 2.

10:15 — 10:40 · Arquitetura + Node + MVC (slides 4–7)

  • Slide 4 (Arquitetura). Desenhar PWA ↔ Node ↔ SQLite no projetor. Apontar os 2 contratos: HTTP/JSON (PWA→Node) e SQL (Node→SQLite). Esse é o mapa-mãe da aula.
  • Slide 5 (Node.js). Event loop, npm, JS dos dois lados. Server HTTP em 8 linhas. Apontar autoestudo "Como sair do zero em Node.js".
  • Slide 6 (SWEBOK v4 Cap. 2). Structural design (camadas) vs behavioral design (fluxos). Cada parte tem 1 motivo para mudar.
  • Slide 7 (MVC no BrPec). Model (evento.model.js) · Controller (sync.controller.js) · Routes (server.js). Mostrar arquivos no editor projetado. Reforçar: Model não conhece HTTP; Controller não escreve SQL.

10:40 — 11:00 · PWA + sync (slides 8–13)

  • Slide 8. 3 pilares PWA em uma página — revisita compacta. Manifest mostrado em snippet.
  • Slide 9. SW lifecycle (install/activate/fetch) + estratégias (cache-first para shell, network-first para /sync/*).
  • Slide 10. sql.js + IndexedDB — boot em 3 passos, db.export() + idbSet() a cada mutação.
  • Slide 11. UUID v4 no cliente. Justificar pedagogicamente.
  • Slides 12–13. Backend Node + idempotência. Mostrar os 3 arquivos MVC e o INSERT OR IGNORE aterrando.

11:00 — 11:20 · TDD + SDD (slides 14–15)

  • Slide 14 (TDD). Ciclo RED → GREEN → REFACTOR. Rodar o spec ao vivo no terminal projetado.
  • Sugestão visual: rm -rf node_modules + npm test → crash; npm install → 5 verdes. RED → GREEN em 30 segundos.
  • Slide 15 (TDD WDIAGW + SDD). Resumir os 2 papers/talks: (1) Ian Cooper — testar porta, não entranha; (2) Spec-Driven Development — onde a indústria está indo.

11:20 — 11:20 (encaixe) · Quizzes socráticos (slides 17–19)

  • Conforme o tempo, intercalar os 3 quizzes (UUID · MVC · TDD bem feito) — 2 min cada — antes ou intercalados ao demo.
  • Formato: pergunta projetada · 30 s silêncio · 2–3 voluntários · revelar resposta-âncora (botão no slide) · amarrar com artefato real.
  • Notas em notas-do-professor/aula-5-perguntas-socraticas.md.
Atenção operacional

Conferir antes da aula: (1) npm install rodou ok no brpec-sync-server (better-sqlite3 às vezes precisa de compilação); (2) Chrome com DevTools fluente; (3) sandbox da aula 4 e PWA da aula anterior ainda funcionais (a PWA usa ../brpec/brpec.db como seed).

Lab cooperativo nos 5 grupos (11:20 — 12:00)

40 minutos. Cada grupo recebe um papel concreto que complementa os outros — não são tarefas independentes. No fim, juntando os 5 prints, a turma tem o ciclo completo documentado.

Bônus universal

Qualquer grupo que terminar primeiro: ler o sync.controller.js e propor 1 validação de payload adicional (ex.: rejeitar tipo fora de {pesagem, vacinação, tratamento, nascimento, mudanca, abate}). Não implementar — só desenhar no caderno. Esse é o aquecimento para a aula 6 (validação + erros estruturados).

Estratégias de metodologia ativa

  • Lab cooperativo — os 5 grupos não fazem a mesma coisa; cada um faz um passo do ciclo. Juntar os prints monta a documentação completa em sala.
  • Predição antes da execução — antes de rodar o sync pela segunda vez, o professor pergunta: "quantos eventos vocês acham que vão entrar?" Força articular idempotência antes de ver.
  • RED → GREEN visível ao vivo — rm -rf node_modules + npm test → crash; npm install → 5 verdes. Em 30 segundos.
  • Quizzes socráticos (3) — sondar concepções (UUID, MVC, TDD bem feito) antes de validar. Resposta-âncora aparece com clique no slide.
  • Aula 5 referencia aula 4 explicitamente — o SQL do Model é o mesmo SQL que aprenderam ontem; muda quem digita.
  • Janela para futuro (SDD) — Spec-Driven Development como horizonte, não como tarefa. Mostra que a aula não termina nela mesma.

Conexão com a aula 6

A aula 6 (Back-End II — endpoints e documentação) expande o servidor de hoje:

  • De POST /sync/deltas + GET /sync/snapshot para CRUD HTTP completo (POST, PUT, DELETE) sobre as entidades do BrPec.
  • Validação de payload com erros estruturados (400/422) — AppError + middleware.
  • Documentação automática (OpenAPI/Swagger ou alternativa minimalista).
  • Como a PWA consome esses endpoints quando online — refresh do snapshot, atualização local.
Para o autoestudo da aula 6

(1) Deixar o sync server rodando local. (2) Listar 3 endpoints CRUD que faltam (ex.: GET /eventos/:bovino_id, PUT /eventos/:uuid, DELETE /eventos/:uuid). (3) Ler material aula 6 sobre validação + AppError. (4) Concluir o autoestudo "Como sair do zero em Node.js em apenas uma aula" se ainda não fez.
